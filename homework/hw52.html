🏷️ 数据标注、查询语言与 Neo4j 工具链综合训练

🏷️ 选择 + 简答 + 代码🏷️ 55 题📖 相关讲义：数据标注、查询语言与 Neo4j 工具链

55 道练习，按“Doccano 标注项目 → 指南、双标与裁决 → 数据划分和导出合同 → SQL/SPARQL/Gremlin/Cypher 选型 → Neo4j 属性图 → 约束、索引与幂等写入 → 参数化、安全和运维边界”组织。做题时始终区分三层责任：平台负责流程，标注规范负责语义，数据库约束与应用代码负责数据完整性和访问安全。

建议先独立作答，再展开参考答案。本练习依据课程“项目应用工具、Doccano、知识查询语言、Neo4j 图数据库”与站内重组讲义设计，并参考 Doccano 官方工作流、W3C SPARQL 1.1、Apache TinkerPop Gremlin、Neo4j Cypher 参数、MERGE和约束文档校准。注意：Doccano 能管理项目、成员、标签和导出，但不会自动替代标注指南与专家裁决；MERGE 只有配合稳定键和约束才适合并发、可重跑导入；当前 Cypher 可用专门的动态语法引用参数化 Label/Relationship Type，但 MATCH (n:$label)、[:$type] 仍是无效写法，封闭医学 Schema 中仍应优先使用审核过的白名单模板并锁定版本。10 道代码题只使用 Python 标准库离线运行，不连接真实数据库。

一、选择题（30 题）

1. Doccano 在医学知识图谱项目中的主要角色是？

A. 组织语料导入、标签配置、成员协作、标注与结果导出
B. 自动定义医学实体边界和事实真伪
C. 代替 Neo4j 存储全部线上知识
D. 自动保证训练集没有泄漏

✅ 参考答案

答案：A。 Doccano 是标注工作流平台。标签语义、边界规则、否定和历史事实是否标注、分歧如何裁决，仍需项目团队在指南和治理流程中明确。

2. 大规模标注前先让两名标注员独立试标同一小批样本，最重要的目的是什么？

A. 暴露指南歧义并估计一致性，再通过裁决修订规范
B. 让同一条样本永久保留两个互相矛盾的答案
C. 只比较谁标得更快
D. 直接把试标集当最终测试集

✅ 参考答案

答案：A。 试标是规范调试，不只是人员考试。应保存原标注、分歧类别、裁决结果和指南版本，使后续质量问题可以追溯。

3. 下列哪一项最应该写入实体标注的“边界规则”？

A. “严重持续性咳嗽”中是否包含程度和持续性修饰语
B. Doccano 页面背景颜色
C. 标注员电脑品牌
D. Neo4j 节点显示尺寸

✅ 参考答案

答案：A。 边界规则必须用正例、反例和困难例说明修饰语、否定词、单位、并列结构与嵌套实体的处理方式，不能只给一个标签名称。

4. 文本写成“咳嗽常见于某疾病”，关系标注应如何确定方向？

A. 按 Schema 固定为 Disease→HAS_SYMPTOM→Symptom，而不是跟随词序
B. 永远从句首指向句尾
C. 随机选择方向
D. 同时生成两条相反语义的边

✅ 参考答案

答案：A。 关系方向属于 Schema。文本语序变化不应改变图中语义；标注工具里的 head/tail 含义也必须在指南中写清。

5. “患者否认发热”最稳妥的处理是？

A. 按指南标记否定状态或排除为肯定事实，不能直接生成 HAS_SYMPTOM
B. 只要出现“发热”就写入肯定症状关系
C. 删除整个患者记录
D. 自动改写为“确诊发热”

✅ 参考答案

答案：A。 mention 的出现不等于事实成立。否定、假设、条件、家族史和既往史应有明确的断言与时间规则，避免把文本共现误写成医学知识。

6. 某标签占 99%，两名标注员都把几乎所有样本标成该标签，为什么只报告总体一致率不够？

A. 类别极不平衡时高一致率可能来自多数类，应结合逐类指标、混淆矩阵和适当的一致性统计
B. 一致率越高一定越差
C. 类别比例不影响任何指标
D. 只需要统计标注速度

✅ 参考答案

答案：A。 分类可结合 Cohen's kappa 等统计，序列标注还应比较实体级 span/label 的 Precision、Recall、F1，并分解边界错和类别错。

7. 同一病历被切成多个相似句子，随机按句切分训练集与测试集的主要风险是？

A. 同源内容跨集合造成泄漏，使测试指标虚高
B. 测试集一定变得更大
C. 所有标签都会消失
D. Cypher 无法查询

✅ 参考答案

答案：A。 应在去重后按患者、文档、病例或来源分组切分。分组键取决于部署时真正独立的样本单位，还要避免使用可识别患者身份的信息作为公开字段。

8. 标注指南从 v1 修改为 v2，并改变了 Disease 的边界规则，正确做法是？

A. 记录版本影响，抽查或回标旧数据，并在数据集元数据中保存指南版本
B. 不留记录直接覆盖标签含义
C. 只改网页标题
D. 假设旧数据自动符合新规则

✅ 参考答案

答案：A。 标签语义变化会形成概念漂移。训练数据、模型、评测结果和导出文件应能追溯到对应指南版本及裁决批次。

9. 标注真实医疗文本时，哪项安全措施最基本？

A. 最小化数据、去标识化、限制成员权限并审计导出
B. 把原始病历放到公开演示项目
C. 在代码中硬编码数据库密码
D. 让所有成员共享管理员账号

✅ 参考答案

答案：A。 标注便利性不能覆盖隐私、授权和合规要求。还应控制下载、日志、备份、保留期限与删除流程，具体要求以所在机构和适用法规为准。

10. NER 导出合同至少应明确什么？

A. 文档 ID、原文、start/end 偏移、标签、标注版本及偏移采用的字符计数规则
B. 只保存标签颜色
C. 只保存实体文本而不保存位置
D. 只保存标注员昵称

✅ 参考答案

答案：A。 不同系统可能按 Unicode 字符、UTF-16 code unit 或字节计偏移。导出后应验证 text[start:end] 与 mention 一致，防止中文、emoji 等导致错位。

11. 哪个场景更适合优先使用 SQL？

A. 查询标注任务表中每位成员的完成量与平均耗时
B. 对 RDF 本体执行标准三元组模式匹配
C. 在 Neo4j 中表达多跳属性图路径
D. 编写 Gremlin 遍历步骤链

✅ 参考答案

答案：A。 成员、任务、状态和日志天然适合关系表及聚合。使用知识图谱不等于把所有运营数据都强行迁入图数据库。

12. SPARQL 的主要查询对象是？

A. RDF 图及其三元组模式
B. 只能是 CSV 单表
C. PyTorch Tensor
D. Doccano 页面组件

✅ 参考答案

答案：A。 SPARQL 是 W3C 的 RDF 查询规范，适合标准本体、Linked Data 与 RDF store；它与 Neo4j 属性图上的 Cypher 不是同一种数据模型接口。

13. Gremlin 最典型的思维方式是？

A. 用一系列 map、filter、side-effect 等步骤组合图遍历
B. 只写二维表 JOIN
C. 只描述 HTML 样式
D. 只能创建 RDF literal

✅ 参考答案

答案：A。 Gremlin 是 Apache TinkerPop 的图遍历语言，可面向多种 TinkerPop 图系统。具体引擎支持、方言差异与执行能力仍需查对应版本文档。

14. Cypher 的核心表达风格是？

A. 用节点、关系和箭头构成图模式并声明要匹配或更新的结果
B. 必须手写每一步磁盘访问
C. 只能执行统计检验
D. 只能在浏览器中使用

✅ 参考答案

答案：A。 例如 (d:Disease)-[:HAS_SYMPTOM]->(s:Symptom) 直接表达有向模式。Cypher 是声明式语言，数据库规划器决定具体执行计划。

15. 在 (d:Disease {id: $id})-[r:HAS_SYMPTOM]->(s:Symptom) 中，Disease 是什么？

A. 节点 Label
B. 关系变量
C. 参数值
D. 属性名

✅ 参考答案

答案：A。 d、r、s 是查询变量，Disease 和 Symptom 是节点标签，HAS_SYMPTOM 是关系类型，id 是属性键，$id 是参数。

16. 何时更适合把“药品生产企业”建成节点而非 Drug 的字符串属性？

A. 企业需要稳定身份、别名、多个药品连接及独立查询时
B. 任何字符串都必须成为节点
C. 企业永远不会复用时
D. 为了增加节点数量

✅ 参考答案

答案：A。 属性还是节点由身份、复用、关系和查询需求决定。Label/relationship/property 的选择应源自业务问题，而非图形是否好看。

17. 关于 CREATE 与 MERGE，哪项正确？

A. CREATE 每次创建；MERGE 先匹配完整模式，不存在时再创建
B. 二者永远完全等价
C. MERGE 自动推断全局唯一 ID
D. CREATE 会自动去重

✅ 参考答案

答案：A。 MERGE 的匹配键和模式必须谨慎设计。把易变属性放入模式会导致属性变化时无法匹配旧节点，从而产生新节点。

18. 下列哪条节点导入更适合可重跑更新？

A. MERGE (d:Disease {id: row.id}) SET d.name = row.name
B. MERGE (d:Disease {id: row.id, name: row.name, updated_at: datetime()})
C. CREATE (d:Disease) 且不保存 ID
D. 每次导入前无条件清空整库

✅ 参考答案

答案：A。 MERGE 模式只放稳定识别键，易变字段在 SET、ON CREATE SET 或 ON MATCH SET 中更新。时间戳放进匹配模式几乎必然破坏幂等性。

19. REQUIRE d.id IS UNIQUE 对缺少 id 属性的 Disease 节点意味着什么？

A. 唯一性约束防重复值，但不一定要求每个节点都存在该属性；存在性需键约束或额外规则
B. 缺失 id 一定自动生成
C. 节点会自动改成 Symptom
D. 所有属性都必须唯一

✅ 参考答案

答案：A。 当前 Neo4j 文档明确区分唯一性、存在性、类型与键约束，且部分约束受版本或版本类型影响。项目应写清所用版本和可用约束，并用验收查询补足业务完整性。

20. 唯一约束与普通索引最准确的区别是？

A. 约束表达并强制数据规则，索引主要帮助定位；某些约束会拥有支持索引
B. 索引自动验证所有医学语义
C. 约束只改变界面颜色

32. 设计一个“试标—一致性检查—专家裁决—扩标”的可审计流程。

✅ 参考答案

33. 医学 NER 应如何报告标注员一致性？

✅ 参考答案

至少报告实体级 exact-match Precision、Recall、F1，并按标签拆分；另统计 span 边界一致但标签不同、标签相同但边界不同、完全漏标等错误。若采用 token 级指标应说明 BIO/BIOES 和分词规则，但不能只用 token accuracy，因为大量 O 会掩盖实体错误。分类任务可补充混淆矩阵与适当的一致性统计。

34. 为什么“先随机切句，再去重”会泄漏？应怎样修正？

✅ 参考答案

同一病历、模板或重复句可能先被分到不同集合，之后在各集合内部去重仍保留跨集合近重复。应先做规范化、精确/近重复检测，确定患者、病例、文档或来源 group，再按 group 切分；切分后再次做跨集合相似性审计。测试集必须模拟真实部署中的独立单位。

35. Doccano 导出后应执行哪些自动校验？

✅ 参考答案

检查文档 ID 唯一、原文哈希、偏移范围、text[start:end] 与 mention、一致的字符计数、允许标签、span 重叠规则、关系端点存在、关系类型和方向符合 Schema、重复标注、空文本、指南版本、来源 group 及跨集合泄漏。失败记录应隔离并给出可定位错误，不能静默修复后丢失原数据。

36. 分别给 SQL、SPARQL、Gremlin、Cypher 一个合适的项目场景。

✅ 参考答案

SQL：统计标注任务、成员进度、审核状态和服务日志；SPARQL：查询 RDF 本体或与标准 Linked Data 互操作；Gremlin：在 TinkerPop 兼容图系统中组合可移植遍历；Cypher：在 Neo4j 属性图中匹配 Disease—HAS_SYMPTOM→Symptom 等模式。选型基于数据模型、互操作、引擎和团队生态，不是按语言“新旧”排序。

37. 如何判断某个字段应建成 Neo4j 节点、关系还是属性？

✅ 参考答案

有独立身份、会被多个对象复用、需要别名或继续连接的对象适合节点；两个节点之间有方向和业务语义的联系适合关系，来源、置信度、有效期可作为关系属性；只描述单个节点且不需独立查询/复用的标量适合属性。还要考虑稳定 ID、基数、更新频率、溯源和主要查询路径。

38. 解释为什么 MERGE (d:Disease {id, name, updated_at}) 容易产生重复，并给出改法。

✅ 参考答案

MERGE 匹配的是整个给定模式，名称修改或每次变化的时间戳都会使旧节点不匹配。应先为 Disease.id 建唯一/键约束，再用 MERGE (d:Disease {id: row.id})，把名称等可变字段放到 SET，把 created_at 放到 ON CREATE SET，把 updated_at 放到 ON MATCH SET 或统一更新逻辑中。

39. 在已有百万节点的库上新增唯一约束，应怎样上线？

✅ 参考答案

先锁定版本和备份策略，在只读或影子环境扫描缺失 ID、重复 ID 与类型错误，输出冲突清单并治理；评估约束创建对全量扫描、锁和资源的影响，安排维护窗口；先在同规模副本演练，再创建具名约束并监控状态；应用需能处理约束冲突。不能期待约束自动合并存量重复实体。

40. 把用户输入“慢支' MATCH (n) DETACH DELETE n //”用于疾病查询，安全接口应怎样设计？

✅ 参考答案

应用先做实体链接得到稳定疾病 ID，再选择审核过的只读查询模板，将 ID 作为 $disease_id 参数传给官方 Driver；数据库账号只授予必要读权限，并设置超时、结果上限和审计。不得把原字符串拼进 Cypher，更不能让用户选择任意破坏性语句。参数化解决值注入，但不替代权限和模板治理。

41. 如何处理动态 Label/Relationship Type 的版本差异？

✅ 参考答案

先锁定 Neo4j/Cypher 主版本并用该版本官方文档和集成测试确认语法。静态位置的 (n:$label)、[:$type] 不合法；较新版本支持专门的动态引用表达式，但封闭医学 Schema 更适合把业务意图映射到少量固定模板或白名单。若采用动态语法，也要限制值、检查计划和兼容性，不能把任意用户文本当结构标识符。

42. 一个按 Disease.id 查询的 Cypher 很慢，如何诊断？

✅ 参考答案

先确认查询结果正确且参数类型一致，再用 EXPLAIN 查看计划、用受控环境的 PROFILE 查看实际行数和算子；检查 Disease.id 的唯一约束/索引是否在线、是否从选择性高的模式开始、是否出现笛卡尔积、无界变长路径或过早聚合；比较估计行数与实际行数，修改后用真实分布数据回归延迟和结果。

43. 为什么事务重试要求写入逻辑幂等？

✅ 参考答案

网络或集群故障可能让客户端不知道事务是否已提交，Driver 也可能重试可重试错误。若每次用 CREATE 或追加不可去重事件，重试会产生重复。应使用稳定 ID、唯一约束、MERGE、事务级业务键和可重复的属性更新；外部副作用需要单独的幂等键或 outbox，不能假设数据库回滚会撤销已发送消息。

44. Neo4j 生产凭据与权限应如何管理？

✅ 参考答案

凭据通过环境变量、密钥服务或运行平台注入，不写进源码、Notebook、网页和日志；开发、导入、只读问答使用不同账号与最小权限；限制网络入口和数据库选择，定期轮换并审计异常查询；错误日志脱敏。Neo4j Browser、Doccano 管理后台和 API 都不应直接暴露到不可信网络。

45. 说明 Doccano、Flask/FastAPI、Gunicorn、Neo4j Browser 和 Neo4j 各自边界。

✅ 参考答案

Doccano 管标注流程；Flask/FastAPI 定义 HTTP 接口与业务校验；Gunicorn 等进程管理器负责运行 WSGI worker，但不自动提供认证、限流和完整监控；Neo4j Browser 是开发查询工具，不是终端医疗产品；Neo4j 存储和查询属性图。生产系统还需反向代理、TLS、认证授权、超时、审计、备份、监控与医疗安全提示。

46. 校验实体标注边界

实现 validate_entities：检查标签白名单、偏移范围、mention 与原文切片一致、重复标注，以及本项目规定的“实体 span 不允许重叠”。

✅ 参考答案
Python
def validate_entities(text: str, annotations: list[dict], allowed_labels: set[str]) -> list[str]:
    errors = []
    valid_spans = []
    seen = set()
    for index, item in enumerate(annotations):
        label = item.get("label")
        start, end = item.get("start"), item.get("end")
        if label not in allowed_labels:
            errors.append(f"{index}: unknown_label")
        if not isinstance(start, int) or not isinstance(end, int) or not (0 <= start < end <= len(text)):
            errors.append(f"{index}: invalid_offset")
            continue
        key = (start, end, label)
        if key in seen:
            errors.append(f"{index}: duplicate")
        seen.add(key)
        if item.get("mention") != text[start:end]:
            errors.append(f"{index}: mention_mismatch")
        valid_spans.append((start, end, index))

    valid_spans.sort()
    for previous, current in zip(valid_spans, valid_spans[1:]):
        if current[0] < previous[1]:
            errors.append(f"{current[2]}: overlaps_{previous[2]}")
    return sorted(errors)


text = "患者否认发热，既往糖尿病。"
fever = text.index("发热")
diabetes = text.index("糖尿病")
annotations = [
    {"start": fever, "end": fever + 2, "label": "Symptom", "mention": "发热"},
    {"start": diabetes, "end": diabetes + 3, "label": "Disease", "mention": "糖尿病"},
]
assert validate_entities(text, annotations, {"Disease", "Symptom"}) == []
bad = annotations + [
    {"start": fever, "end": fever + 2, "label": "Symptom", "mention": "高热"},
    {"start": -1, "end": 2, "label": "Unknown", "mention": "患者"},
]
errors = validate_entities(text, bad, {"Disease", "Symptom"})
assert "2: duplicate" in errors and "2: mention_mismatch" in errors
assert "3: invalid_offset" in errors and "3: unknown_label" in errors
print(errors)

47. 按 Schema 归一关系方向

输入关系的 head/tail 可能跟随文本语序。根据端点实体类型，把它统一成 Schema 方向；类型无法匹配时拒绝。

✅ 参考答案
Python
SCHEMA = {"HAS_SYMPTOM": ("Disease", "Symptom")}


def normalize_relation(relation: dict, entities: dict[str, dict]) -> dict:
    head, tail, relation_type = relation["head"], relation["tail"], relation["type"]
    if head not in entities or tail not in entities:
        raise ValueError("关系端点不存在")
    expected = SCHEMA.get(relation_type)
    if expected is None:
        raise ValueError("未知关系类型")
    head_type, tail_type = entities[head]["label"], entities[tail]["label"]
    if (head_type, tail_type) == expected:
        source, target = head, tail
    elif (tail_type, head_type) == expected:
        source, target = tail, head
    else:
        raise ValueError("端点类型不符合 Schema")
    return {"source": source, "type": relation_type, "target": target}


entities = {
    "e1": {"label": "Disease", "mention": "慢性支气管炎"},
    "e2": {"label": "Symptom", "mention": "咳嗽"},
    "e3": {"label": "Drug", "mention": "某药"},
}
assert normalize_relation({"head": "e2", "tail": "e1", "type": "HAS_SYMPTOM"}, entities) == {
    "source": "e1", "type": "HAS_SYMPTOM", "target": "e2"
}
try:
    normalize_relation({"head": "e3", "tail": "e2", "type": "HAS_SYMPTOM"}, entities)
except ValueError as error:
    assert str(error) == "端点类型不符合 Schema"
else:
    raise AssertionError("应拒绝非法关系")
print("ok")

48. 按来源分组进行确定性数据切分

同一 group_id 的句子必须进入同一集合。使用稳定哈希实现 80%/10%/10% 切分，并验证不存在跨集合 group 泄漏。

✅ 参考答案
Python
import hashlib


def group_split(records: list[dict], ratios=(0.8, 0.1, 0.1)) -> dict[str, list[dict]]:
    if len(ratios) != 3 or any(r < 0 for r in ratios) or abs(sum(ratios) - 1.0) > 1e-9:
        raise ValueError("ratios 必须是和为 1 的三个非负数")
    result = {"train": [], "validation": [], "test": []}
    first_cut, second_cut = ratios[0], ratios[0] + ratios[1]
    for record in records:
        group_id = str(record["group_id"])
        digest = hashlib.sha256(group_id.encode("utf-8")).digest()
        score = int.from_bytes(digest[:8], "big") / 2**64
        split = "train" if score < first_cut else "validation" if score < second_cut else "test"
        result[split].append(record)
    return result


records = [
    {"id": "s1", "group_id": "doc:A"},
    {"id": "s2", "group_id": "doc:A"},
    {"id": "s3", "group_id": "doc:B"},
    {"id": "s4", "group_id": "doc:C"},
]
split = group_split(records)
location = {}
for split_name, rows in split.items():
    for row in rows:
        previous = location.setdefault(row["group_id"], split_name)
        assert previous == split_name
assert split == group_split(records)
print({name: [r["id"] for r in rows] for name, rows in split.items()})

49. 计算实体级严格匹配 Precision、Recall 与 F1

只有文档、start、end、label 全部相同才算命中；重复预测不得重复计分。

✅ 参考答案
Python
def entity_metrics(gold: list[dict], predicted: list[dict]) -> dict[str, float]:
    fields = ("doc_id", "start", "end", "label")
    gold_set = {tuple(item[field] for field in fields) for item in gold}
    predicted_set = {tuple(item[field] for field in fields) for item in predicted}
    true_positive = len(gold_set & predicted_set)
    precision = true_positive / len(predicted_set) if predicted_set else 0.0
    recall = true_positive / len(gold_set) if gold_set else 0.0
    f1 = 2 * precision * recall / (precision + recall) if precision + recall else 0.0
    return {"precision": precision, "recall": recall, "f1": f1}


gold = [
    {"doc_id": "1", "start": 0, "end": 2, "label": "Disease"},
    {"doc_id": "1", "start": 3, "end": 5, "label": "Symptom"},
    {"doc_id": "2", "start": 0, "end": 2, "label": "Drug"},
]
predicted = [
    gold[0], gold[1], gold[1],
    {"doc_id": "2", "start": 0, "end": 2, "label": "Disease"},
]
metrics = entity_metrics(gold, predicted)
assert round(metrics["precision"], 4) == 0.6667
assert round(metrics["recall"], 4) == 0.6667
assert round(metrics["f1"], 4) == 0.6667

50. 生成双标分歧报告

对同一文档的两份实体标注，区分“相同 span 但标签不同”和“只在一方出现”。

✅ 参考答案
Python
def disagreement_report(a: list[dict], b: list[dict]) -> dict:
    def index_by_span(items):
        result = {}
        for item in items:
            key = (item["doc_id"], item["start"], item["end"])
            if key in result:
                raise ValueError("同一标注员存在重复 span")
            result[key] = item["label"]
        return result

    a_index, b_index = index_by_span(a), index_by_span(b)
    common = set(a_index) & set(b_index)
    label_mismatch = [
        {"span": key, "a": a_index[key], "b": b_index[key]}
        for key in sorted(common) if a_index[key] != b_index[key]
    ]
    return {
        "label_mismatch": label_mismatch,
        "only_a": sorted(set(a_index) - set(b_index)),
        "only_b": sorted(set(b_index) - set(a_index)),
    }


a = [
    {"doc_id": "1", "start": 0, "end": 2, "label": "Disease"},
    {"doc_id": "1", "start": 3, "end": 5, "label": "Symptom"},
]
b = [
    {"doc_id": "1", "start": 0, "end": 2, "label": "Symptom"},
    {"doc_id": "1", "start": 6, "end": 8, "label": "Drug"},
]
report = disagreement_report(a, b)
assert len(report["label_mismatch"]) == 1
assert report["only_a"] == [("1", 3, 5)]
assert report["only_b"] == [("1", 6, 8)]
print(report)

52. 为动态关系类型实现程序白名单路由

RELATION_QUERIES = {
    "HAS_SYMPTOM": """
MATCH (d:Disease {id: $source_id})
MATCH (s:Symptom {id: $target_id})
MERGE (d)-[r:HAS_SYMPTOM]->(s)
SET r.source = $source
""".strip(),
    "RECOMMEND_DRUG": """
MATCH (d:Disease {id: $source_id})
MATCH (x:Drug {id: $target_id})
MERGE (d)-[r:RECOMMEND_DRUG]->(x)
SET r.source = $source
""".strip(),
}


def route_relation(row: dict) -> tuple[str, dict]:
    relation_type = row.get("type")

    params = {
        "source_id": row["source_id"],
        "target_id": row["target_id"],
        "source": row["evidence_source"],
    }
    return RELATION_QUERIES[relation_type], params

53. 模拟基于稳定 ID 的节点 MERGE

实现内存版 merge_node：首次创建保留 created_at，再次导入更新白名单属性；同一 ID 类型冲突时拒绝。

✅ 参考答案
Python
from copy import deepcopy


UPDATABLE = {"name", "source", "updated_at"}


def merge_node(store: dict[str, dict], node_id: str, label: str, properties: dict) -> bool:
    if not node_id or not label:
        raise ValueError("id 和 label 不能为空")
    forbidden = set(properties) - (UPDATABLE | {"created_at"})
    if forbidden:
        raise ValueError(f"不允许更新字段: {sorted(forbidden)}")
    if node_id not in store:
        store[node_id] = {"id": node_id, "label": label}
        store[node_id].update(deepcopy(properties))
        return True
    node = store[node_id]
    if node["label"] != label:
        raise ValueError("同一稳定 ID 出现类型冲突")
    for key in UPDATABLE:
        if key in properties:
            node[key] = deepcopy(properties[key])
    return False


store = {}
created = merge_node(store, "disease:D001", "Disease", {
    "name": "慢支", "source": "v1", "created_at": "2026-01-01", "updated_at": "2026-01-01"
})
created_again = merge_node(store, "disease:D001", "Disease", {
    "name": "慢性支气管炎", "source": "v2", "created_at": "SHOULD_NOT_REPLACE", "updated_at": "2026-02-01"
})
assert created is True and created_again is False
assert len(store) == 1
assert store["disease:D001"]["created_at"] == "2026-01-01"
assert store["disease:D001"]["name"] == "慢性支气管炎"
print(store)

54. 为 UNWIND 批量关系写入准备去重参数

校验端点存在和 Schema 类型，按三元组合并重复来源，输出可直接作为 $rows 的参数列表。

✅ 参考答案
Python
from collections import defaultdict


SCHEMA = {"HAS_SYMPTOM": ("Disease", "Symptom")}


def prepare_relation_rows(entities: dict[str, dict], claims: list[dict]) -> list[dict]:
    grouped = defaultdict(set)
    for claim in claims:
        source_id, relation_type, target_id = claim["source_id"], claim["type"], claim["target_id"]
        if source_id not in entities or target_id not in entities:
            raise ValueError("存在悬空关系")
        expected = SCHEMA.get(relation_type)
        actual = (entities[source_id]["label"], entities[target_id]["label"])
        if expected != actual:
            raise ValueError("关系端点类型或方向错误")
        if not claim.get("evidence_source"):
            raise ValueError("缺少证据来源")
        grouped[(source_id, relation_type, target_id)].add(claim["evidence_source"])
    return [
        {"source_id": key[0], "target_id": key[2], "sources": sorted(sources)}
        for key, sources in sorted(grouped.items())
    ]


entities = {"d:1": {"label": "Disease"}, "s:1": {"label": "Symptom"}}
claims = [
    {"source_id": "d:1", "type": "HAS_SYMPTOM", "target_id": "s:1", "evidence_source": "a"},
    {"source_id": "d:1", "type": "HAS_SYMPTOM", "target_id": "s:1", "evidence_source": "b"},
    {"source_id": "d:1", "type": "HAS_SYMPTOM", "target_id": "s:1", "evidence_source": "a"},
]
rows = prepare_relation_rows(entities, claims)
assert rows == [{"source_id": "d:1", "target_id": "s:1", "sources": ["a", "b"]}]
cypher = """
UNWIND $rows AS row
MATCH (d:Disease {id: row.source_id})
MATCH (s:Symptom {id: row.target_id})
MERGE (d)-[r:HAS_SYMPTOM]->(s)
SET r.sources = row.sources
""".strip()
assert "$rows" in cypher
print(cypher, {"rows": rows})

55. 完成“标注导出验收 → 关系归一 → Neo4j 写入计划”最小闭环

实现 dry-run：校验实体切片和关系端点，按 Schema 归一方向，生成固定的节点/关系 Cypher 及参数；发现否定关系时不入图并记录原因。

✅ 参考答案
Python
NODE_QUERY = """
UNWIND $rows AS row
MERGE (n:Entity {id: row.id})
SET n.name = row.name, n.entity_type = row.label
""".strip()

REL_QUERY = """
UNWIND $rows AS row
MATCH (d:Entity {id: row.source_id})
MATCH (s:Entity {id: row.target_id})
MERGE (d)-[r:HAS_SYMPTOM]->(s)
SET r.annotation_version = row.annotation_version
""".strip()


def make_import_plan(document: dict) -> dict:
    text = document["text"]
    entities = {}
    for item in document["entities"]:
        start, end = item["start"], item["end"]
        if not (0 <= start < end <= len(text)) or text[start:end] != item["mention"]:
            raise ValueError("实体偏移或 mention 错误")
        if item["id"] in entities:
            raise ValueError("实体 ID 重复")
        entities[item["id"]] = item

    relation_rows, skipped = [], []
    for relation in document["relations"]:
        if relation["head"] not in entities or relation["tail"] not in entities:
            raise ValueError("关系端点不存在")
        if relation.get("assertion") != "affirmed":
            skipped.append({"relation": relation, "reason": "not_affirmed"})
            continue
        head, tail = entities[relation["head"]], entities[relation["tail"]]
        if relation["type"] != "HAS_SYMPTOM":
            raise ValueError("关系类型不在本模板白名单")
        if (head["label"], tail["label"]) == ("Disease", "Symptom"):
            source, target = head, tail
        elif (tail["label"], head["label"]) == ("Disease", "Symptom"):
            source, target = tail, head
        else:
            raise ValueError("关系端点类型错误")
        relation_rows.append({
            "source_id": source["id"], "target_id": target["id"],
            "annotation_version": document["annotation_version"],
        })

    node_rows = [
        {"id": item["id"], "name": item["mention"], "label": item["label"]}
        for item in sorted(entities.values(), key=lambda x: x["id"])
    ]
    return {
        "node_write": {"query": NODE_QUERY, "params": {"rows": node_rows}},
        "relation_write": {"query": REL_QUERY, "params": {"rows": relation_rows}},
        "skipped": skipped,
    }


text = "慢性支气管炎常见咳嗽，但患者否认发热。"
disease_start = text.index("慢性支气管炎")
cough_start = text.index("咳嗽")
fever_start = text.index("发热")
document = {
    "text": text,
    "annotation_version": "guide:v2",
    "entities": [
        {"id": "d:1", "start": disease_start, "end": disease_start + len("慢性支气管炎"),
         "label": "Disease", "mention": "慢性支气管炎"},
        {"id": "s:1", "start": cough_start, "end": cough_start + 2,
         "label": "Symptom", "mention": "咳嗽"},
        {"id": "s:2", "start": fever_start, "end": fever_start + 2,
         "label": "Symptom", "mention": "发热"},
    ],
    "relations": [
        {"head": "s:1", "tail": "d:1", "type": "HAS_SYMPTOM", "assertion": "affirmed"},
        {"head": "d:1", "tail": "s:2", "type": "HAS_SYMPTOM", "assertion": "negated"},
    ],
}
plan = make_import_plan(document)
assert plan["relation_write"]["params"]["rows"] == [{
    "source_id": "d:1", "target_id": "s:1", "annotation_version": "guide:v2"

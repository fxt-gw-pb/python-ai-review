🧰 Function Call：工具调用原理与三种定义方式综合训练

🏷️ 选择 + 简答 + 代码🏷️ 40 题📖 相关章节：Function Call：让模型可靠调用工具🚀 所属项目：SmartVoyage：Agent 与多智能体旅行助手

40 道练习，覆盖 Function Call 解决的问题、模型只出参不执行、两轮消息循环，以及 JSON Schema / @tool / Pydantic 三种定义方式与 Agent 封装。重点练「何时该调工具、如何安全分发与回填结果」。

建议先独立作答，再展开参考答案。代码题用纯 Python 模拟工具分发与消息循环，不调用真实大模型 API。

一、选择题（20 题）

1. Function Call 让模型输出的核心形态是？

A. 直接在云端执行任意系统命令
B. 包含调用函数所需参数的结构化对象（如 JSON）
C. 只能输出 Markdown 标题
D. 只能输出图片

✅ 参考答案

答案：B。 课件强调：模型智能选择是否调用，并给出符合参数约定的结构化结果，再由应用侧执行。

2. 课件列出 Function Call 可缓解的三类局限不包括？

A. 信息实时性
B. 数据局限性（专业库/外部 API）
C. 功能扩展性（计算、分析等）
D. 彻底消除所有幻觉且无需权限控制

✅ 参考答案

答案：D。 FC 扩展能力与时效，但不自动等于安全与事实完备；执行侧仍需校验与审计。

3. 关于「模型会不会执行函数」，正确的是？

A. 模型在云端直接跑你的 Python 函数
B. 模型不执行函数，只返回调用所需参数；由开发者在应用中执行
C. 模型只能执行加减法
D. 模型执行后不会再生成文本

✅ 参考答案

答案：B。 这是 Function Call 与普通文本补全的关键边界。

4. 无 Function Call 时，典型链路更接近？

A. 用户 → 服务 → 模型 →（可能多轮纯文本）
B. 用户直接改数据库主键
C. 只下载模型权重
D. 只训练 Tokenizer

✅ 参考答案

答案：A。 课件对比：无工具时模式更简单，主要是提示词往返。

5. 有 Function Call 时，正确顺序是？

A. 先随便执行本地函数，再问模型要不要
B. 把提示与可用函数发给模型 → 模型决定文本或函数调用 → 若调用则服务端执行并把结果回传 → 模型组织最终回答
C. 跳过模型，直接返回空
D. 只返回函数名、从不回填结果

✅ 参考答案

答案：B。 两阶段：选工具/出参，再基于工具结果生成连贯回答。

6. 自定义 tool 的 JSON 中，type 字段通常设为？

A. "table"
B. "function"
C. "css"
D. "gpu"

✅ 参考答案

答案：B。 Schema 根上声明这是函数工具。

7. description 字段对模型的作用是？

A. 仅给人类看，模型忽略
B. 描述函数功能，模型据此决定是否/如何调用
C. 必须等于函数源码
D. 只能写英文缩写

✅ 参考答案

答案：B。 描述质量直接影响工具选择准确率。

8. parameters 应用什么形式准确描述入参？

A. 纯自然语言长文、无结构
B. JSON Schema 对象（properties / required 等）
C. 随机字符串
D. 仅文件路径

✅ 参考答案

答案：B。 无参数时可省略；有参数则用 Schema 约束类型与必填项。

9. llm.bind_tools(tools, tool_choice="auto") 中 "auto" 表示？

A. 禁止一切工具
B. 允许模型自动决定是否调用工具
C. 强制每次调用全部工具
D. 只绑定第一个工具

✅ 参考答案

答案：B。 也可按框架支持强制指定某工具；课件示例用 auto。

10. 判断模型是否发起工具调用，课件代码看的是？

A. 只看 HTTP 状态码
B. ai_msg.tool_calls 是否存在且非空
C. 只看 temperature
D. 只看 GPU 显存

✅ 参考答案

答案：B。 有 tool_calls 则进入执行与回填分支。

11. 工具执行后回传给模型时，消息类型是？

A. 只用 HumanMessage 再发一遍原问题
B. ToolMessage（含 content 与 tool_call_id）
C. 只能发邮件
D. 删除全部历史

✅ 参考答案

答案：B。 用 tool_call_id 对齐某次调用，便于多工具并行/串行。

12. @tool 装饰器的主要优势是？

A. 必须手写全部 JSON Schema
B. 根据函数签名与 docstring 自动生成工具定义，写法简洁
C. 取消类型注解
D. 禁止参数校验

✅ 参考答案

答案：B。 适合快速开发与原型；校验能力相对基础。

13. 使用 @tool 后执行工具时，课件示例调用方式是？

A. selected_tool(**args) 与普通函数完全一样即可
B. selected_tool.invoke(tool_call["args"])
C. 只能 eval
D. 不能执行

✅ 参考答案

答案：B。 装饰后对象按工具 API 的 invoke 调用（与纯函数分发略有不同）。

14. Pydantic 方式定义工具的突出能力是？

A. 更弱的类型检查
B. 更丰富的数据验证与更清晰的类结构，但需手写 invoke
C. 不能描述参数
D. 只能用于前端

✅ 参考答案

答案：B。 适合复杂约束与自定义执行逻辑。

15. 三种定义方式对比中，自动化程度最高的是？

A. 手写 JSON Schema
B. @tool 装饰器
C. 必须汇编
D. 三者相同

✅ 参考答案

答案：B。 课件表：JSON 低、Pydantic 中、@tool 高。

16. 需要与外部系统最大灵活集成、Schema 需精确可控时，更倾向？

A. 只靠口头约定
B. 手写 JSON Schema
C. 删除 tools
D. 随机函数名

✅ 参考答案

答案：B。 通用、灵活，但样板代码多、校验需自理或外挂。

17. 课件对 Agent 的公式化理解是？

A. Agent = 仅向量数据库
B. Agent = 大模型 + 任务规划 + 外部工具 + 记忆
C. Agent = 仅 CSS
D. Agent = 仅微调权重

✅ 参考答案

答案：B。 Function Call 常作为「用工具」的底层机制；框架会封装循环。

18. initialize_agent 相对手写两轮循环的主要价值是？

A. 完全不能调工具
B. 框架封装规划/调用过程，使用更省事，但仍依赖工具定义与模型能力
C. 不需要大模型
D. 禁止 verbose

✅ 参考答案

答案：B。 课件：很多 Agent 框架封装内部调用，更易用。

19. 若工具描述写成「处理各种事情」，最可能的后果是？

A. 工具选择更精准
B. 模型更易误选/乱调该工具
C. 自动提升忠实度到 1
D. 取消 JSON

✅ 参考答案

答案：B。 description 应具体、可区分；模糊描述是误调用主因之一。

20. 安全上，收到 tool_calls 后首先应做的是？

A. 无校验直接 eval 任意代码
B. 白名单校验工具名与参数类型/范围，再执行
C. 把密钥打印到日志公开处
D. 跳过 ToolMessage

✅ 参考答案

答案：B。 模型可能被提示注入诱导调用危险工具；执行侧必须设防。

二、简答题（12 题）

21. 用对比句说明：Function Call 与「让模型在回答里假装算完」的差别。

✅ 参考答案

FC 把计算/查库交给可验证的外部函数，结果再回填生成；纯文本「假装算完」不可靠且难审计。前者可测、可权限控制，后者易幻觉。

22. 用文字描述有工具时的最小消息状态机。

✅ 参考答案

HumanMessage → 模型 →（无 tool）直接最终文本；或 → tool_calls → 应用执行 → ToolMessage(s) → 再 invoke → 最终文本。可多轮直到无新的 tool_calls 或达步数上限。

23. 为什么必须把工具结果用 ToolMessage 绑回同一个 tool_call_id？

✅ 参考答案

一次回复可能含多个并行调用；id 保证结果对齐正确调用。错绑会导致模型用错结果或无法结束工具回合。

24. 比较 JSON Schema / @tool / Pydantic：各举一个更合适的业务场景。

✅ 参考答案
  • JSON：要对齐外部 OpenAPI、跨语言客户端。
  • @tool：内部快速加加减乘除、原型。
  • Pydantic：参数约束复杂（枚举、范围、嵌套）且要强校验。

25. 工具场景里常用较低 temperature 的动机是什么？

✅ 参考答案

降低随机性，使是否调用、参数取值更稳定，便于回归测试。复杂开放生成仍可能提高温度，但工具选择阶段常偏低。

26. 为什么「模型支持 Function Call」仍可能调错工具？

✅ 参考答案

描述不清、工具过多相似、提示冲突、用户注入、缺少 few-shot/约束等都会导致误选。需要评测集、日志与权限边界，而不是只看模型宣传能力。

27. 手写两轮循环与 Agent 框架，调试时各有什么优劣？

✅ 参考答案

手写：每条 message 可见，易定位；但样板多、易漏边界。Agent：省事、支持更复杂规划；但内部封装使排障更依赖 verbose/回调与轨迹导出。

28. required: ["a","b"] 漏写会带来什么风险？

✅ 参考答案

模型可能省略关键参数，应用侧解包失败或静默用默认值。应在 Schema 与执行前双重校验必填项。

29. 若第一次 invoke 没有 tool_calls，课件分支如何处理？

✅ 参考答案

视为直接文本回答：打印/返回 ai_msg.content，不再执行工具。业务上仍可记录「本应调用却未调用」的评测案例。

30. Agent 公式里的 Memory 与单次 Function Call 演示有何不同？

✅ 参考答案

单次演示多为当轮 messages；Memory 强调跨轮状态（对话摘要、中间结果、用户偏好）。复杂任务常需显式记忆策略，避免上下文膨胀。

31. 把密钥写进仓库里的 Config.api_key 有什么问题？应怎么做？

✅ 参考答案

泄露风险高。应使用环境变量/密钥管理，日志脱敏，示例用占位符，权限最小化。

32. 设计一个「禁止直接执行、只允许白名单工具」的检查清单（至少 4 条）。

✅ 参考答案

示例：工具名白名单；参数 JSON Schema 校验；超时与重试；副作用工具需二次确认；审计日志含 call_id/参数摘要；拒绝未知字段。

三、代码题（8 题）

33. 校验函数工具 JSON Schema 的最小结构

要求根上 type=="function"，且含 function.name、function.description、function.parameters。

✅ 参考答案

def validate_tool_schema(tool: dict) -> None:
    if tool.get("type") != "function":
        raise ValueError("type must be function")
    fn = tool.get("function")
    if not isinstance(fn, dict):
        raise ValueError("missing function")
    for key in ("name", "description", "parameters"):
        if key not in fn:
            raise ValueError(f"missing function.{key}")
    if not isinstance(fn["name"], str) or not fn["name"]:
        raise ValueError("invalid name")


ok = {
    "type": "function",
    "function": {
        "name": "add",
        "description": "将数字a与数字b相加",
        "parameters": {"type": "object", "properties": {}, "required": []},
    },
}
validate_tool_schema(ok)
try:
    validate_tool_schema({"type": "function", "function": {"name": "x"}})
    raise AssertionError("should fail")
except ValueError:
    pass
print("schema ok")

34. 实现工具注册表：按名分发并校验必填参数

✅ 参考答案
Python
def make_registry(fns: dict, required: dict):
    def dispatch(name: str, args: dict):
        key = name.lower()
        if key not in fns:
            raise KeyError(f"unknown tool: {name}")
        miss = [k for k in required[key] if k not in args]
        if miss:
            raise ValueError(f"missing args: {miss}")
        return fns[key](**args)
    return dispatch


def add(a, b):
    return a + b


def multiply(a, b):
    return a * b


dispatch = make_registry(
    {"add": add, "multiply": multiply},
    {"add": ["a", "b"], "multiply": ["a", "b"]},
)
assert dispatch("add", {"a": 2, "b": 1}) == 3
try:
    dispatch("add", {"a": 1})
    raise AssertionError("should fail")
except ValueError:
    pass
print(dispatch("Multiply", {"a": 3, "b": 4}))

def run_tools(ai_msg, registry):
    messages = [ai_msg]
    tool_messages = []
    for call in ai_msg.get("tool_calls", []):
        out = registry(call["name"], call["args"])
        tool_messages.append({
            "role": "tool",
            "tool_call_id": call["id"],
            "content": str(out),
        })
    messages.extend(tool_messages)
    return messages


reg = lambda name, args: {"add": lambda **kw: kw["a"] + kw["b"]}[name.lower()](**args)

36. 实现极简「假模型」两轮循环

第一轮若问题含 + 则返回 add 的 tool_call；第二轮根据 ToolMessage 生成最终文本。

✅ 参考答案
Python
def fake_llm(messages, tools_bound=True):
    last = messages[-1]
    if last["role"] == "user" and "+" in last["content"] and tools_bound:
        # very small parser: "2+1"
        import re
        m = re.search(r"(\d+)\s*\+\s*(\d+)", last["content"])
        a, b = int(m.group(1)), int(m.group(2))
        return {
            "role": "assistant",
            "content": "",
            "tool_calls": [{"id": "t1", "name": "add", "args": {"a": a, "b": b}}],
        }
    if last["role"] == "tool":
        return {"role": "assistant", "content": f"结果是 {last['content']}", "tool_calls": []}
    return {"role": "assistant", "content": last.get("content", ""), "tool_calls": []}


def add(a, b):
    return a + b


def chat(query):
    messages = [{"role": "user", "content": query}]
    ai = fake_llm(messages)
    messages.append(ai)
    if ai.get("tool_calls"):
        call = ai["tool_calls"][0]
        out = add(**call["args"])
        messages.append({"role": "tool", "tool_call_id": call["id"], "content": str(out)})
        final = fake_llm(messages)
        messages.append(final)
        return final["content"]
    return ai["content"]


assert chat("2+1等于多少？") == "结果是 3"
print(chat("2+1等于多少？"))

37. 从函数对象生成「伪 @tool schema」（名称、描述、参数名）

不依赖 LangChain：用 __name__、__doc__ 首行与注解生成字典。

✅ 参考答案
Python
import inspect


def tool_schema_from_fn(fn):
    sig = inspect.signature(fn)
    props = {}
    required = []
    for name, param in sig.parameters.items():
        ann = param.annotation
        typ = "integer" if ann is int else "string"
        props[name] = {"type": typ, "description": name}
        if param.default is inspect.Parameter.empty:
            required.append(name)
    doc = (fn.__doc__ or "").strip().splitlines()[0] if fn.__doc__ else fn.__name__
    return {
        "type": "function",
        "function": {
            "name": fn.__name__,
            "description": doc,
            "parameters": {"type": "object", "properties": props, "required": required},
        },
    }


def add(a: int, b: int) -> int:
    """将数字a与数字b相加"""
    return a + b


sch = tool_schema_from_fn(add)
assert sch["function"]["name"] == "add"
assert set(sch["function"]["parameters"]["required"]) == {"a", "b"}
print(sch["function"]["description"])

38. 实现带白名单与确认机制的副作用工具调度

lookup 可直接执行；cancel_order 必须 confirmed=True。

✅ 参考答案
Python
ORDERS = {"A100": "paid"}


def lookup_order(order_id: str) -> str:
    return ORDERS.get(order_id, "not_found")


def cancel_order(order_id: str) -> str:
    if order_id not in ORDERS:
        return "not_found"
    ORDERS[order_id] = "cancelled"
    return "cancelled"


SCHEMAS = {
    "lookup_order": {"fn": lookup_order, "fields": {"order_id": str}, "confirm": False},
    "cancel_order": {"fn": cancel_order, "fields": {"order_id": str}, "confirm": True},
}


def dispatch(call: dict, confirmed: bool = False):
    name = call["name"]
    if name not in SCHEMAS:
        raise PermissionError("tool not allowed")
    spec = SCHEMAS[name]
    args = call.get("args", {})
    if set(args) != set(spec["fields"]):
        raise ValueError("unexpected fields")
    for k, typ in spec["fields"].items():
        if not isinstance(args[k], typ):
            raise TypeError(k)
    if spec["confirm"] and not confirmed:
        raise PermissionError("confirmation required")
    return spec["fn"](**args)


assert dispatch({"name": "lookup_order", "args": {"order_id": "A100"}}) == "paid"
try:
    dispatch({"name": "cancel_order", "args": {"order_id": "A100"}})
    raise AssertionError("should fail")
except PermissionError:
    pass
assert dispatch({"name": "cancel_order", "args": {"order_id": "A100"}}, confirmed=True) == "cancelled"
print(ORDERS)

39. 限制最大工具轮次，防止无限循环

若连续 max_rounds 轮仍产生 tool_calls，则停止并返回错误摘要。

✅ 参考答案
Python
def run_with_limit(user_text, llm, registry, max_rounds=3):
    messages = [{"role": "user", "content": user_text}]
    for _ in range(max_rounds):
        ai = llm(messages)
        messages.append(ai)
        calls = ai.get("tool_calls") or []
        if not calls:
            return {"ok": True, "content": ai.get("content", ""), "messages": messages}
        for call in calls:
            out = registry(call["name"], call["args"])
            messages.append({
                "role": "tool",
                "tool_call_id": call["id"],
                "content": str(out),
            })
    return {"ok": False, "content": "max tool rounds exceeded", "messages": messages}


def noisy_llm(messages):
    # always asks to add 1+1
    return {
        "role": "assistant",
        "content": "",
        "tool_calls": [{"id": "x", "name": "add", "args": {"a": 1, "b": 1}}],
    }


reg = lambda n, a: a["a"] + a["b"]
res = run_with_limit("hi", noisy_llm, reg, max_rounds=2)
assert res["ok"] is False
print(res["content"])

40. 对比三种「定义风格」的元数据完整度（教学用打分）

对同一工具，分别给 JSON / 装饰器风 / Pydantic 风的描述字典，按是否含 name、description、typed params、required、validate 钩子打分。

✅ 参考答案
Python
def score_style(meta: dict) -> int:
    score = 0
    score += 1 if meta.get("name") else 0
    score += 1 if meta.get("description") else 0
    score += 1 if meta.get("typed_params") else 0
    score += 1 if meta.get("required") else 0
    score += 1 if meta.get("validate_hook") else 0
    return score


json_style = {
    "name": "add", "description": "加法", "typed_params": True,
    "required": True, "validate_hook": False,
}
decorator_style = {
    "name": "add", "description": "加法", "typed_params": True,
    "required": True, "validate_hook": False,
}
pydantic_style = {
    "name": "add", "description": "加法", "typed_params": True,
    "required": True, "validate_hook": True,
}
assert score_style(pydantic_style) >= score_style(json_style)
assert score_style(decorator_style) == 4
print(score_style(json_style), score_style(decorator_style), score_style(pydantic_style))

这对应课件对比表的直觉：三种都能表达工具；Pydantic 在「校验钩子/复杂约束」上更强，@tool 在自动化上更强，JSON 在跨系统契约上更通用。

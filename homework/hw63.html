🔌 MCP：模型与外部工具的标准接口综合训练

🏷️ 选择 + 简答 + 代码🏷️ 40 题📖 相关章节：MCP：模型与外部工具的标准接口🚀 所属项目：SmartVoyage：Agent 与多智能体旅行助手

40 道练习，覆盖 MCP 出现的动机、「一次编写到处调用」、Client/Server/Host 角色、五步调用链路，以及 stdio / SSE / Streamable 三种传输与 FastMCP 用法。重点练「工具描述归谁写」和「本地 IPC vs 网络传输」选型。

建议先独立作答，再展开参考答案。代码题用纯 Python 模拟清单发现与调用路由，不启动真实 MCP 进程。

一、选择题（20 题）

1. 课件提出：工具描述不应主要由谁来写？

A. 工具开发者
B. 每次新应用的工具调用方/应用开发者（各自理解不一致）
C. 操作系统内核
D. 显卡驱动

✅ 参考答案

答案：B。 每人描述方式不一导致质量不稳，且重复劳动；应由工具开发者把能力说清楚。

2. MCP 希望达成的工程效果更接近？

A. 每次项目手写全新、互不兼容的工具 JSON
B. 工具自描述，实现「一次编写，到处调用」
C. 取消全部外部工具
D. 只允许本地文件读写

✅ 参考答案

答案：B。 把描述权还给工具侧，客户端做发现与调用。

3. MCP 全称是？

A. Multi Core Processor
B. Model Context Protocol（模型上下文协议）
C. Message Compression Pipeline
D. MySQL Connection Pool

✅ 参考答案

答案：B。 开放协议，用于 LLM 与外部数据源/工具的安全双向连接。

4. 课件把 MCP 类比为 AI 世界的？

A. 仅 USB 线材
B. 「Type-C」式通用接口（愿景上接近 HTTP 的通用程度）
C. 仅 JPEG 格式
D. 仅 GPU 驱动

✅ 参考答案

答案：B。 强调标准化连接工具与数据。

5. MCP 服务端（Tool Provider）的核心职责是？

A. 只负责训练大模型
B. 把本地函数包装成标准接口，监听请求、执行并返回结果
C. 只渲染前端页面
D. 只存储 CSS

✅ 参考答案

答案：B。 例如天气、数学、数据库访问等服务。

6. MCP 客户端（Tool Consumer）的核心职责是？

A. 代替服务端执行 SQL
B. 连接服务端、自发现工具列表，并按需调用
C. 只能打印日志
D. 禁止连接多个 Server

✅ 参考答案

答案：B。 典型消费者：Agent、自动化脚本、需远程能力的应用。

7. 「MCP Host」与「MCP Client」的关系，课件说法是？

A. 二者完全同义且可互换删除
B. Host 是发起请求的 LLM 应用；Client 是主机内部的对象
C. Host 只能是数据库
D. Client 只能是浏览器插件商店

✅ 参考答案

答案：B。 应用（Host）内部持有 Client 去连若干 Server。

8. 工具清单（Tool Manifest）里通常包含？

A. 仅 GPU 温度
B. name、description、input_schema（及可能的 output_schema）等
C. 仅仓库 star 数
D. 仅 CSS 颜色

✅ 参考答案

答案：B。 Client 缓存后上报给 LLM，使其「知道」可用工具。

9. 五步流程中，决定「调用哪个工具」的通常是？

A. 随机数生成器
B. LLM 根据用户语义选择，并生成 function calling 指令
C. 仅操作系统调度器
D. 仅 DNS

✅ 参考答案

答案：B。 之后才由 Client 匹配 Server 真正执行。

10. MCP Client 执行调用时，真正跑业务逻辑的位置是？

A. 只在浏览器 localStorage
B. MCP Server 内部执行工具逻辑
C. 必须在用户手机短信里
D. 只能在 PDF 里

✅ 参考答案

答案：B。 Client 负责协议路由；Server 负责执行。

11. 结果回传给 LLM 时，课件对应到应用消息层的形态是？

A. 删除全部历史
B. 包装为 ToolMessage 再送回模型生成最终自然语言
C. 只能发邮件
D. 必须重启电脑

✅ 参考答案

答案：B。 与 Function Call 回填思路一致，只是工具可能来自远程 MCP Server。

12. 课件强调：传输方式与 MCP 协议本身？

A. 完全等同，换传输就换协议语义
B. 无关；stdio / SSE / Streamable 是通信承载
C. 只能使用 UDP 广播
D. 禁止本地通信

✅ 参考答案

答案：B。 协议定义交互语义；传输可按部署选择。

13. stdio 传输的典型特征是？

A. 必须跨公网
B. 客户端拉起服务端子进程，经 stdin/stdout 通信的本地 IPC
C. 只能单向推送
D. 不能封装命令行工具

✅ 参考答案

答案：B。 简单高效、无网络开销，适合本地快速封装脚本。

14. SSE 更准确的描述是？

A. 纯本地管道、无 HTTP
B. 基于 HTTP 的单向推送（服务器持续向客户端发事件流）
C. 只能 UDP
D. 必须双向 WebRTC

✅ 参考答案

答案：B。 适合远端部署、网络访问；课件对比表标为单向。

15. 相对 SSE，Streamable-HTTP 课件强调的差异包括？

A. 只能本地 IPC
B. 传输格式更灵活，通信可为双向流
C. 完全不能用于网络
D. 已被 USB 取代

✅ 参考答案

答案：B。 适合复杂实时双向流场景。

16. 关于早期「HTTP + SSE」组合，课件提示是？

A. 永远是唯一推荐方式
B. 在规范中已标注弃用，2025-03-26 起更推荐 Streamable HTTP
C. 只用于训练 YOLO
D. 与 MCP 无关

✅ 参考答案

答案：B。 复习时以现行推荐为准，旧教程可能仍写 HTTP+SSE。

17. FastMCP 服务端示例用 @mcp.tool(...) 后，stdio 启动调用是？

A. mcp.run(transport="stdio")
B. print("hello") 即可对外服务
C. 必须手写 TCP socket 字节协议
D. 只能 eval

✅ 参考答案

答案：A。 装饰器注册工具，run 指定传输。

18. stdio 客户端课件注意点：直接跑客户端时？

A. 必须先手动另开终端永久挂起服务端，否则绝对无法工作
B. 客户端可按配置自动拉起服务端子进程；编码问题可用 UTF-8 环境变量兜底
C. 禁止使用 asyncio
D. 不能 call_tool

✅ 参考答案

答案：B。 课件：直接启动客户端即可；必要时设置 PYTHONIOENCODING=utf-8。

19. load_mcp_tools(session) 的作用更接近？

A. 训练新模型
B. 从 session 拉取 Server 工具并适配成可给 Agent/LangChain 使用的工具列表
C. 删除全部工具
D. 只压缩图片

✅ 参考答案

答案：B。 体现「自发现」：不必在应用里手写死全部 Schema。

20. Function Call 与 MCP 的分工，更合理的说法是？

A. 二者互相排斥，有 MCP 就禁止 FC
B. FC 是模型侧结构化决策；MCP 是工具暴露与跨应用复用的标准连接层
C. MCP 只能做加减法
D. FC 只能跑在服务端、MCP 只能跑在浏览器

✅ 参考答案

答案：B。 流程上常是：LLM 出 FC → Client 经 MCP 调 Server → 回填。

二、简答题（12 题）

21. 为什么「工具调用方各自写描述」会导致质量不稳？

✅ 参考答案

同一工具被不同人用不同粒度/术语描述，模型选工具与填参行为漂移；还造成每个新应用重复劳动。工具作者最清楚边界、参数与失败模式。

22. 用一句话区分 MCP Server / Client / Host。

✅ 参考答案

Server 提供并执行工具；Client 负责发现与调用；Host 是承载 Client 的 LLM 应用本身。

23. 按课件顺序写出 MCP 工具调用五步，并标明每步产物。

✅ 参考答案
  1. 连接 Server → 拿到 Tool Manifest；
  2. LLM 决策 → FC JSON（name/arguments）；
  3. Client 路由 → 协议请求发到对应 Server；
  4. Server 执行 → result；
  5. Client 回填 ToolMessage → LLM 最终自然语言。

24. 「自发现」相对手写死 tools 列表的两个好处与一个风险。

✅ 参考答案

好处：少重复、工具升级后客户端可刷新清单。风险：不可信 Server 可能暴露危险工具，必须鉴权、白名单与审计。

25. 什么时候优先选 stdio，而不是 SSE/Streamable？

✅ 参考答案

工具与 Agent 同机、要快速封装本地 CLI/脚本、希望零网络配置与更小攻击面时。多机共享、远程团队访问则倾向网络传输。

26. SSE「单向」在 MCP 实践里意味着什么限制？如何理解课件仍用它做请求响应？

✅ 参考答案

事件流语义上是服务器推客户端；早期远程方案常配合客户端 HTTP POST。规范演进后更推荐 Streamable HTTP 做更自然的双向远程通信。

27. 对比表里 stdio 的「可靠性」来源与网络方案有何不同？

✅ 参考答案

stdio 依赖本地 OS 管道语义；SSE/Streamable 通常跑在 TCP 之上由传输层保证字节可靠送达。但仍要处理应用层超时、半包与进程崩溃。

28. FastMCP 里 name 与 description 写差会怎样？

✅ 参考答案

与 FC 一样：模型难选中或乱选；跨应用复用时文档漂移。应把参数约束、副作用、失败返回写清楚，并做调用回归。

29. session.initialize() 之后再 call_tool 的原因？

✅ 参考答案

完成握手/能力协商与会话就绪；未初始化就调用易失败或不稳定。发现工具也应在会话就绪后进行。

30. 为什么说 MCP 把「描述权」还给工具开发者后，应用开发者仍要做安全治理？

✅ 参考答案

标准化降低集成成本，但不自动保证工具无害。仍需 Server 信任模型、权限、参数校验、超时、审计与最小权限密钥。

31. python_a2a 示例把 MCP 挂到 FastAPI/uvicorn 的动机是什么？

✅ 参考答案

用 HTTP 服务形态暴露工具，便于远端/多客户端访问与和 A2A 生态组合；与纯 stdio 本地拉起形成互补。

32. 设计一个「引入 MCP 前后」的重复劳动对比（各两点）。

✅ 参考答案

引入前：每应用手写 Schema；描述不一致难维护。引入后：Server 统一清单；多 Host 共享同一工具实现与升级。

三、代码题（8 题）

33. 校验 Tool Manifest 条目

每项必须含非空 name、description，且 input_schema 为 dict。

✅ 参考答案
Python
def validate_manifest(items: list[dict]) -> None:
    if not items:
        raise ValueError("empty manifest")
    names = []
    for it in items:
        name = it.get("name")
        desc = it.get("description")
        schema = it.get("input_schema")
        if not isinstance(name, str) or not name.strip():
            raise ValueError("bad name")
        if not isinstance(desc, str) or not desc.strip():
            raise ValueError("bad description")
        if not isinstance(schema, dict):
            raise ValueError("bad input_schema")
        names.append(name)
    if len(names) != len(set(names)):
        raise ValueError("duplicate tool names")


validate_manifest([
    {"name": "get_weather", "description": "查询天气", "input_schema": {"type": "object"}},
])
try:
    validate_manifest([{"name": "x", "description": "", "input_schema": {}}])
    raise AssertionError("should fail")
except ValueError:
    pass
print("manifest ok")

34. 实现内存版 MCP Server：注册工具并执行

✅ 参考答案
Python
class MiniMCPServer:
    def __init__(self, name: str):
        self.name = name
        self._tools = {}

    def tool(self, name: str, description: str):
        def deco(fn):
            self._tools[name] = {"description": description, "fn": fn}
            return fn
        return deco

    def manifest(self):
        return [
            {"name": n, "description": m["description"], "input_schema": {"type": "object"}}
            for n, m in self._tools.items()
        ]

    def call(self, name: str, args: dict | None = None):
        if name not in self._tools:
            raise KeyError(name)
        return self._tools[name]["fn"](**(args or {}))


srv = MiniMCPServer("demo")

@srv.tool("get_weather", "查询天气")
def get_weather():
    return "北京的天气是多云"

assert "get_weather" in [x["name"] for x in srv.manifest()]
assert srv.call("get_weather") == "北京的天气是多云"
print(srv.manifest())

35. 实现 Client：连接多个 Server，合并清单并按名路由调用

✅ 参考答案

class MiniMCPClient:
    def __init__(self):
        self.servers = {}  # tool_name -> server

    def connect(self, server):
        for item in server.manifest():
            name = item["name"]
            if name in self.servers:
                raise ValueError(f"tool name conflict: {name}")
            self.servers[name] = server

    def list_tools(self):
        out = []
        for name, srv in self.servers.items():
            meta = next(x for x in srv.manifest() if x["name"] == name)
            out.append(meta)
        return out

    def call_tool(self, name: str, arguments: dict | None = None):
        if name not in self.servers:
            raise KeyError(name)
        return self.servers[name].call(name, arguments)


class S:
    def __init__(self, tools):
        self._tools = tools
    def manifest(self):
        return [{"name": n, "description": d, "input_schema": {}} for n, (d, _) in self._tools.items()]
    def call(self, name, args=None):
        return self._tools[name][1](**(args or {}))


weather = S({"get_weather": ("查询天气", lambda: "多云")})
faq = S({"query_faq": ("查 FAQ", lambda: "恐龙？")})
client = MiniMCPClient()
client.connect(weather)
client.connect(faq)
assert {t["name"] for t in client.list_tools()} == {"get_weather", "query_faq"}
assert client.call_tool("query_faq") == "恐龙？"
print(client.list_tools())

def pick_transport(profile: str) -> str:
    mapping = {
        "local_cli": "stdio",
        "remote_unidirectional": "sse",
        "remote_bidirectional": "streamable",
    }
    if profile not in mapping:
        raise ValueError(profile)
    return mapping[profile]

assert pick_transport("remote_bidirectional") == "streamable"

class StdioLauncher:
    def __init__(self, command: str, args: list[str]):
        self.command = command
        self.args = args
        self.running = False
        self.server = None

    def start(self, server):
        # 教学：真实实现会 Popen；这里只标记运行并挂上 server
        self.running = True
        self.server = server

    def call(self, tool: str, arguments=None):
        if not self.running or self.server is None:
            raise RuntimeError("server not started")
        return self.server.call(tool, arguments)

39. 冲突检测：两个 Server 暴露同名工具时拒绝连接

✅ 参考答案

class Registry:
    def __init__(self):
        self.owner = {}

    def register(self, server_name: str, tools: list[str]):
        for t in tools:
            if t in self.owner and self.owner[t] != server_name:
                raise ValueError(f"conflict on {t}: {self.owner[t]} vs {server_name}")
            self.owner[t] = server_name


r = Registry()
r.register("weather", ["get_weather"])
try:
    r.register("other", ["get_weather"])
    raise AssertionError("should fail")
except ValueError as e:
    assert "conflict" in str(e)
print(r.owner)

40. 端到端迷你 Host：发现工具 → 决策 → 调用 → 生成回答，并记录审计日志

✅ 参考答案

class Host:
    def __init__(self, client):
        self.client = client
        self.audit = []

    def available(self):
        return [t["name"] for t in self.client.list_tools()]

    def handle(self, user_text: str):
        tools = self.available()
        # 极简策略：关键词命中工具名片段
        chosen = None
        if "天气" in user_text and "get_weather" in tools:
            chosen = "get_weather"
        elif "问题" in user_text and "query_faq" in tools:
            chosen = "query_faq"
        if not chosen:
            ans = "无匹配工具"
            self.audit.append({"tool": None, "result": None, "answer": ans})
            return ans
        result = self.client.call_tool(chosen, {})
        ans = f"{chosen} => {result}"
        self.audit.append({"tool": chosen, "result": result, "answer": ans})
        return ans


class S:
    def __init__(self, mapping):
        self.mapping = mapping
    def manifest(self):
        return [{"name": n, "description": n, "input_schema": {}} for n in self.mapping]
    def call(self, name, args=None):
        return self.mapping[name]()

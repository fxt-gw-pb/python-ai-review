🔎 BM25：稀疏检索原理与实现综合训练

🏷️ 选择 + 简答 + 代码🏷️ 40 题📖 相关章节：BM25：稀疏检索原理与实现🚀 所属项目：EduRAG：企业私有知识智能问答

40 道练习，覆盖 BM25 相对 TF-IDF 的改进、TF/IDF/长度归一化与 $k_1$、$b$、中文分词检索链路，以及 BM25Search 工程封装。重点练「稀疏关键词匹配何时够用、何时要接向量检索」。

建议先独立作答，再展开参考答案。代码题用纯 Python 实现公式思想，不依赖 jieba / rank_bm25（语料已预分词）。

一、选择题（20 题）

1. BM25 的全称 Best Matching 25 在信息检索中主要用于？

A. 训练深度神经网络权重
B. 计算查询与文档的相关性得分并排名
C. 压缩图片文件
D. 管理关系型事务锁

✅ 参考答案

答案：B。 它是经典稀疏检索打分函数，对 query 与每篇文档给出可排序的相关分。

2. 相对传统 TF-IDF，BM25 明确引入的两项改进是？

A. 卷积与池化
B. 文档长度归一化与词频饱和
C. 注意力掩码与位置编码
D. 梯度裁剪与学习率衰减

✅ 参考答案

答案：B。 词频不再线性无限放大；长文档优势被长度项抑制，检索更稳。

3. 公式中的 $f_i$ 表示？

A. 语料中的文档总数
B. 词项 $q_i$ 在文档 $D$ 中的出现次数（TF）
C. 平均文档长度
D. 平滑参数 $k_1$

✅ 参考答案

答案：B。 课件记法：$f_i$ 是词项在文档内的频次。

4. 公式中的 $N$ 与 $n_i$ 分别是？

A. $N$ 为词表大小，$n_i$ 为向量维度
B. $N$ 为文档总数，$n_i$ 为包含词项 $q_i$ 的文档数
C. 二者都是 $k_1$
D. 二者都是字符数

✅ 参考答案

答案：B。 IDF 侧用「稀有词加权」：$n_i$ 越小，通常区分度越高。

5. $|D|$ 与 $\mathrm{avgdl}$ 的作用是？

A. 只用于打印日志
B. 刻画当前文档长度相对语料平均长度，服务长度归一化
C. 替换所有 TF
D. 强制每篇文档等长

✅ 参考答案

答案：B。 参数 $b$ 控制长度惩罚强度；$\mathrm{avgdl}$ 是语料级统计量。

6. 课件给出的常用参数默认值是？

A. $k_1=0$，$b=1$
B. $k_1=1.2$，$b=0.75$
C. $k_1=100$，$b=0$
D. 两者必须相等

✅ 参考答案

答案：B。 这是经典经验默认；业务上仍可用验证集微调。

7. 词频饱和机制的直观含义是？

A. 词出现次数翻倍，相关分也必须严格翻倍
B. 词频升高带来的增益会逐渐变缓，避免「堆词」无限抬分
C. 禁止任何词出现超过一次
D. 只统计首字母

✅ 参考答案

答案：B。 $k_1$ 越大，饱和越慢；过小则很快忽略额外重复。

8. 将 $b$ 设为 0 时，长度归一化项会？

A. 仍然强制按最长文档惩罚
B. 基本关闭长度惩罚，打分不再依赖 $|D|/\mathrm{avgdl}$
C. 删除 IDF
D. 使所有文档得分为负

✅ 参考答案

答案：B。 $b=0$ 时长度因子退化；$b=1$ 时长度影响最强（具体形式以实现公式为准）。

9. 课件简单示例中，查询「他喜欢编程」相对文档「我喜欢编程」「编程很有趣」，更可能因哪些词重叠抬高第一篇？

A. 只有「他」
B. 「喜欢」与「编程」的重叠
C. 只有标点
D. 只有停用词「很」

✅ 参考答案

答案：B。 查询与文档1共享「喜欢」「编程」；文档2主要共享「编程」。实际库实现（如 BM25L）细节会影响数值，但词重叠是主因。

10. 课件示例运行日志给出的最佳匹配是？

A. 「编程很有趣」
B. 「我喜欢编程」
C. 空文档
D. 随机一篇

✅ 参考答案

答案：B。 日志写明：查询「他喜欢编程」，最佳匹配「我喜欢编程」，得分约 1.094。若文中「分析」段与此矛盾，应以可复现输出与公式为准做核对。

11. 中文场景使用 jieba.lcut 的主要原因是？

A. BM25 只能接受英文空格分词，中文必须先切词
B. 为了训练 CNN
C. 为了替换 Redis
D. 为了生成向量维度

✅ 参考答案

答案：A。 BM25 工作在「词项序列」上；中文无空格时需分词（或字/子词方案），分词质量直接影响召回。

12. 课件检索模块选用的是 rank_bm25 中的？

A. 仅 TF-IDF
B. BM25L
C. 仅 HNSW
D. 仅余弦距离

✅ 参考答案

答案：B。 from rank_bm25 import BM25L；同库还有 Okapi 等变体，API 类似但饱和/长度细节不同。

13. BM25Search.__init__ 中正确的初始化顺序是？

A. 先 search 再分词
B. 保存文档 → 分词得到 tokenized_docs → 用其构造 BM25 模型
C. 先删日志再随机打分
D. 不需要分词

✅ 参考答案

答案：B。 模型建立在「已分词语料」上；语料变更通常需重建索引/模型。

14. search 方法中 scores.argmax() 的含义是？

A. 返回最低分文档
B. 返回得分最高的文档下标
C. 返回词表大小
D. 返回平均文档长度

✅ 参考答案

答案：B。 再取 documents[best_idx] 与对应分数；异常时课件返回 (None, 0.0)。

15. 在 EduRAG 的 FAQ 路径里，BM25 更像哪类能力？

A. 稠密向量语义检索
B. 稀疏关键词相关匹配（适合标准问法/字面重合）
C. 图像超分
D. 强化学习对齐

✅ 参考答案

答案：B。 与 Milvus 稠密检索互补：FAQ/关键词命中走稀疏，长尾语义走向量 + 生成。

16. 下列哪项不是 BM25 稀疏检索的典型优势？

A. 可解释：命中哪些词较直观
B. 对专有名词、型号、错误拼写变体的鲁棒性通常强于一切稠密模型
C. 实现轻、可快速对标准问打分
D. 不依赖 GPU

✅ 参考答案

答案：B。 稀疏法对同义改写、语序变化往往弱；那是稠密检索更擅长的地方。专有名词场景稀疏常强，但不是「强于一切稠密」。

17. 依赖文件只需列出？

A. torch 与 pymilvus 即可
B. jieba 与 rank_bm25
C. 只需要 fastapi
D. 不需要任何依赖

✅ 参考答案

答案：B。 课件 requirements.txt 即此二者（工程里还会有日志等标准库）。

18. 若查询词在所有文档中都出现（$n_i=N$），IDF 贡献通常会？

A. 变得极大，压过一切 TF
B. 变得很小/接近无区分度，难以靠该词分辨文档
C. 强制得分为 1
D. 删除该词的长度归一化

✅ 参考答案

答案：B。 停用词或普遍词 IDF 低；真正抬分的是「在查询中出现、但只在少数文档出现」的词。

19. 文档极长且反复堆砌查询词时，BM25 相对朴素 TF 的更合理行为是？

A. 分数随堆词近似线性爆炸
B. 词频饱和 + 长度惩罚抑制「靠长度和重复刷分」
C. 直接把长文档设为无限分
D. 忽略所有查询词

✅ 参考答案

答案：B。 这也是课件总结强调的两点。

20. 将 BM25 最高分与阈值（如 0.85）比较再路由 RAG，工程上还需要注意？

A. 不同实现/语料下分数量纲不同，阈值不可照搬，需标定
B. 任何语料阈值都永恒等于 0.85
C. 阈值只对向量检索有意义
D. 不需要日志

✅ 参考答案

答案：A。 BM25 绝对分依赖分词、参数与语料；应用侧应用验证集/线上抽样标定阈值，并打日志。

二、简答题（12 题）

21. 用一句话对比 TF-IDF 与 BM25。

✅ 参考答案

TF-IDF 用词频与逆文档频率衡量重要性；BM25 在此基础上加入词频饱和与文档长度归一化，减轻「堆词」和「长文优势」，排名通常更稳。

22. 解释 $k_1$ 过大或过小可能带来的检索现象。

✅ 参考答案

$k_1$ 过小：词频很快饱和，重复出现几乎不再加分；过大：更接近线性 TF，堆词文档更容易被抬高。应在业务查询集上看是否过拟合「关键词刷屏」。

23. 解释 $b$ 过大或过小可能带来的检索现象。

✅ 参考答案

$b$ 偏大：更惩罚长文档，短而精的文档更易胜出；$b$ 偏小：长度影响弱，长文档即使稀释了主题也可能靠绝对词频占优。需结合语料长度分布调节。

24. 为什么中文 FAQ 匹配里分词错误比「公式写错一位小数」更致命？

✅ 参考答案

BM25 的基本单位是词项。错切（如专有名词被拆碎、该合的未合）会改变 TF/IDF 统计，导致该命中的词对不上。应沉淀领域词典并回归测试高频问法。

25. 课件目录为何拆成 retrieval/bm25_search.py 与 main.py？

✅ 参考答案

检索能力模块化：模型初始化、打分、日志与异常在检索类内；入口只负责语料与演示调用。后续可替换为连 MySQL 的 FAQ 检索而不改业务主流程。

26. 描述 get_scores → argmax 这条链路的输出语义与风险。

✅ 参考答案

对语料每篇文档打一个相关分，取最高者作为「最佳匹配」。风险：最高分仍可能很低（全都不相关）；或并列高分时 argmax 只取一个。工程上常加阈值、返回 top-k 并人工/规则复核。

27. 为何说「分析段写文档2更高」与「运行日志文档1更高」冲突时该怎么处理？

✅ 参考答案

以可复现实验与公式为准：核对分词、库变体（BM25L/Okapi）、随机性与文本是否一致。文档笔误常见；复习时应自己跑通并修正笔记，而不是死记矛盾结论。

28. BM25 与稠密向量检索如何互补？各举一个更适合的 query。

✅ 参考答案

BM25：字面重合强，如「退费多久到账」「错误码 E1234」。稠密：同义改写/语义相近，如「钱什么时候能回来」。EduRAG 先 FAQ/BM25 短路，再走 Milvus RAG。

29. 若语料新增大量文档，为什么往往要重建 BM25 模型？

✅ 参考答案

$N$、$n_i$、$\mathrm{avgdl}$ 等统计量会变，IDF 与长度归一化随之变。增量策略可以有，但简单实现通常整体重分词 + 重初始化更安全。

30. 日志在 BM25 模块里应至少打哪些字段便于排障？

✅ 参考答案

原始 query、分词结果、top 分数与文档摘要/id、阈值判定路径、异常栈。没有分词与分数，很难区分「阈值问题」还是「切词问题」。

31. 停用词要不要一律删除？结合 BM25 谈利弊。

✅ 参考答案

删除可降噪、缩词表；但某些场景停用词参与固定表达（如「如何」「是否」）。一刀切可能伤 FAQ。更稳是：领域停用词表 + 评测，而不是盲目套用通用表。

32. 把 BM25 得分直接当「概率」或「余弦相似度」有什么问题？

✅ 参考答案

量纲与上界依赖实现和语料，未必在 $[0,1]$，也不能直接与向量相似度比大小。跨系统融合要校准、归一或分路阈值，不能裸比数值。

35. 实现 BM25 单词项贡献（含饱和与长度归一）

$$ \frac{f(k_1+1)}{f + k_1\bigl(1-b+b\cdot|D|/\mathrm{avgdl}\bigr)}\cdot \mathrm{IDF} $$

✅ 参考答案
Python
import math


def idf(N, n_i):
    return math.log(1.0 + (N - n_i + 0.5) / (n_i + 0.5))


def bm25_term(f, dl, avgdl, N, n_i, k1=1.2, b=0.75):
    denom = f + k1 * (1 - b + b * dl / avgdl)
    return (f * (k1 + 1) / denom) * idf(N, n_i)


# same TF, longer doc should score lower when b>0
s_short = bm25_term(1, dl=3, avgdl=3, N=2, n_i=1)
s_long = bm25_term(1, dl=30, avgdl=3, N=2, n_i=1)
assert s_short > s_long
print(s_short, s_long)

36. 实现简易 BM25 文档得分：对查询词项求和

未知词（$n_i=0$）仍用 IDF 公式；文档内 TF 为 0 的词贡献为 0。

✅ 参考答案
Python
import math
from collections import Counter


def idf(N, n_i):
    return math.log(1.0 + (N - n_i + 0.5) / (n_i + 0.5))


def avg_dl(docs):
    return sum(len(d) for d in docs) / max(len(docs), 1)


def bm25_score(query, doc, docs, k1=1.2, b=0.75):
    N = len(docs)
    avgdl = avg_dl(docs)
    df = Counter(term for d in docs for term in set(d))
    tfc = Counter(doc)
    dl = len(doc)
    score = 0.0
    for term in query:
        f = tfc[term]
        if f == 0:
            continue
        n_i = df[term]
        denom = f + k1 * (1 - b + b * dl / avgdl)
        score += (f * (k1 + 1) / denom) * idf(N, n_i)
    return score


docs = [["我", "喜欢", "编程"], ["编程", "很", "有趣"]]
q = ["他", "喜欢", "编程"]
s0 = bm25_score(q, docs[0], docs)
s1 = bm25_score(q, docs[1], docs)
assert s0 > s1  # overlaps 喜欢+编程 vs only 编程
print(s0, s1)

37. 封装 SimpleBM25：建库、打分、返回最佳匹配

接口对齐课件：search(query_tokens) -> (doc_text, score)；空语料返回 (None, 0.0)。

✅ 参考答案
Python
from collections import Counter
import math


class SimpleBM25:
    def __init__(self, documents, tokenized_docs):
        if len(documents) != len(tokenized_docs):
            raise ValueError("length mismatch")
        self.documents = list(documents)
        self.tokenized_docs = [list(d) for d in tokenized_docs]
        self.N = len(self.tokenized_docs)
        self.avgdl = (
            sum(len(d) for d in self.tokenized_docs) / self.N if self.N else 0.0
        )
        self.df = Counter(t for d in self.tokenized_docs for t in set(d))

    def _score_one(self, query, doc, k1=1.2, b=0.75):
        tfc = Counter(doc)
        dl = len(doc)
        score = 0.0
        for term in query:
            f = tfc[term]
            if f == 0:
                continue
            n_i = self.df[term]
            idf = math.log(1.0 + (self.N - n_i + 0.5) / (n_i + 0.5))
            denom = f + k1 * (1 - b + b * dl / self.avgdl)
            score += (f * (k1 + 1) / denom) * idf
        return score

    def search(self, query):
        if not self.documents:
            return None, 0.0
        scores = [self._score_one(query, d) for d in self.tokenized_docs]
        best_idx = max(range(len(scores)), key=lambda i: (scores[i], -i))
        return self.documents[best_idx], scores[best_idx]


docs = ["我喜欢编程", "编程很有趣"]
toks = [["我", "喜欢", "编程"], ["编程", "很", "有趣"]]
eng = SimpleBM25(docs, toks)
best, score = eng.search(["他", "喜欢", "编程"])
assert best == "我喜欢编程" and score > 0
print(best, round(score, 3))

38. 实现 top-k 检索与阈值门控

返回按分降序的 top-k；若最高分 < threshold 则 path="rag"，否则 path="faq"。

✅ 参考答案
Python
def rank_topk(engine, query, k=2):
    scored = []
    for doc, toks in zip(engine.documents, engine.tokenized_docs):
        scored.append((doc, engine._score_one(query, toks)))
    scored.sort(key=lambda x: (-x[1], x[0]))
    return scored[:k]


def route(engine, query, threshold=0.8):
    ranked = rank_topk(engine, query, k=1)
    if not ranked:
        return {"path": "rag", "doc": None, "score": 0.0}
    doc, score = ranked[0]
    if score >= threshold:
        return {"path": "faq", "doc": doc, "score": score}
    return {"path": "rag", "doc": None, "score": score}

40. 实现「预分词缓存」：避免每次 search 重新切文档

CorpusIndex 在初始化时缓存 tokenized_docs；update(doc) 追加文档并刷新 DF/avgdl；batch_scores(query) 返回全部分数列表。

✅ 参考答案
Python
import math
from collections import Counter


class CorpusIndex:
    def __init__(self, docs_tokens):
        self.tokenized_docs = [list(d) for d in docs_tokens]
        self._recompute()

    def _recompute(self):
        self.N = len(self.tokenized_docs)
        self.avgdl = (
            sum(len(d) for d in self.tokenized_docs) / self.N if self.N else 0.0
        )
        self.df = Counter(t for d in self.tokenized_docs for t in set(d))

    def update(self, doc_tokens):
        self.tokenized_docs.append(list(doc_tokens))
        self._recompute()

    def batch_scores(self, query, k1=1.2, b=0.75):
        out = []
        for doc in self.tokenized_docs:
            tfc = Counter(doc)
            dl = len(doc)
            s = 0.0
            for term in query:
                f = tfc[term]
                if f == 0:
                    continue
                n_i = self.df[term]
                idf = math.log(1.0 + (self.N - n_i + 0.5) / (n_i + 0.5))
                denom = f + k1 * (1 - b + b * dl / self.avgdl)
                s += (f * (k1 + 1) / denom) * idf
            out.append(s)
        return out


idx = CorpusIndex([["我", "喜欢", "编程"], ["编程", "很", "有趣"]])
s1 = idx.batch_scores(["喜欢", "编程"])
idx.update(["喜欢", "音乐"])
s2 = idx.batch_scores(["喜欢", "编程"])
assert len(s1) == 2 and len(s2) == 3
assert s2[0] != s1[0]  # N/DF/avgdl 变化后旧文档分数也会变
print(s1, s2)

🗃️ Neo4j 医学图谱搭建与质量验证综合训练

🏷️ 选择 + 简答 + 代码🏷️ 55 题📖 相关讲义：Neo4j 医学图谱搭建与质量验证

55 道练习，按“原始 JSON → 节点/关系中间层 → 约束与索引 → 参数化批量写入 → 幂等与增量 → 图质量查询 → 回滚、安全和医学边界”组织。做题时始终记住：建图不是把数据塞进 Neo4j，而是把可验证、可追溯的数据合同安全地发布为可重复构建的图。

建议先独立作答，再展开参考答案。本练习依据原课件第六章和站内重组讲义设计，并用 Neo4j 官方的 约束语法、MERGE 语义、UNWIND 语义、Python Driver 查询参数及事务与重试校准。10 道代码题只使用 Python 标准库模拟导入前的数据治理与验收，不连接真实数据库，不访问真实患者数据。

一、选择题（30 题）

1. 将 medical.json 建成医学知识图谱时，最稳妥的第一步是什么？

A. 先生成并验证节点、关系中间记录，再连接数据库 \ B. 先清空生产数据库 \ C. 直接把每行 JSON 拼进 Cypher \ D. 只看 Neo4j Browser 是否出现彩色节点

✅ 参考答案

答案：A。 中间层将解析、标准化与数据库写入解耦，便于单元测试、对账、失败重放和回滚。图可视化只是展示，不是数据正确性的证明。

2. 原始字段名为 recommand_drug，而图 Schema 计划使用 recommend_drug，合理做法是？

A. 在输入适配层兼容旧字段，在规范化中间层统一为新字段并记录版本 \ B. 把拼写错误永久扩散到所有接口 \ C. 同时创建两个语义完全相同的关系类型 \ D. 静默丢弃该字段

✅ 参考答案

答案：A。 原始快照应保持可追溯，规范化层负责字段映射。映射规则和 Schema 版本进入 manifest，避免上下游各自猜测字段含义。

3. 为什么不应直接用疾病名称作为唯一主键？

A. 名称存在别名、同名异义、纠错和多语言变化 \ B. Neo4j 节点不能保存字符串 \ C. 名称不能建立索引 \ D. 稳定 ID 只能是自增整数

✅ 参考答案

答案：A。 名称是可变属性，不是概念身份。优先使用上游标准 ID；缺少标准 ID 时也要通过受控映射表分配持久 ID，而不是每次按当前名称重新生成。

4. 下列哪条最符合当前 Cypher 的单属性唯一约束写法？

A. CREATE CONSTRAINT disease_id_unique IF NOT EXISTS FOR (n:Disease) REQUIRE n.id IS UNIQUE \ B. CREATE UNIQUE Disease.id \ C. ALTER NODE Disease UNIQUE id \ D. CREATE INDEX Disease(id) UNIQUE

✅ 参考答案

答案：A。 IF NOT EXISTS 使 Schema 初始化可重复执行；唯一约束同时阻止同一标签下重复 id，并由索引支撑查找。

5. 普通 name 索引与 id 唯一约束的关键区别是？

A. 普通索引主要加速查询，不保证值唯一；唯一约束同时校验唯一性 \ B. 普通索引会自动删除重复节点 \ C. 唯一约束只能用于关系 \ D. 两者语义完全相同

✅ 参考答案

答案：A。 name 常用于检索但允许同名；id 用于身份，应由约束守住唯一性。不要把“查得快”和“数据合法”混为一谈。

6. 在可重复执行的建图任务中，CREATE 与 MERGE 的主要差别是？

A. CREATE 总是新建，MERGE 尝试匹配完整模式，未匹配才创建 \ B. CREATE 会自动去重 \ C. MERGE 永远不会创建数据 \ D. MERGE 不支持属性

✅ 参考答案

答案：A。 MERGE 类似“匹配或创建”，但只有配合稳定键和约束才能可靠控制重复；它不是自动理解业务实体的魔法去重器。

7. 哪种节点写法最适合幂等导入？

A. MERGE (n:Disease {id: row.id})，再 SET 可变属性 \ B. MERGE (n:Disease {id: row.id, name: row.name, updated_at: datetime()}) \ C. CREATE (n:Disease {name: row.name}) \ D. MERGE (n {name: row.name})

✅ 参考答案

答案：A。 MERGE 模式只放稳定身份键。若把 name 或当前时间放进去，属性变化会使完整模式无法匹配，从而产生意外节点或约束冲突。

8. 合并 Disease 到 Symptom 的关系时，推荐先 MATCH 两端节点再 MERGE 关系，主要因为？

A. 可以确保端点已按各自稳定 ID 绑定，避免整段模式意外创建节点 \ B. MATCH 会自动补全缺失节点 \ C. 关系不能有属性 \ D. MERGE 只能用于节点

✅ 参考答案

答案：A。 对已绑定的端点执行 MERGE (d)-[r:HAS_SYMPTOM]->(s)，身份边界最清楚。若端点不存在，本行不应偷偷造一个信息不完整的新节点，而应进入异常队列。

9. 哪个属性最适合只在 ON CREATE SET 中赋值？

A. created_at \ B. updated_at \ C. 最新显示名称 \ D. 最近一次导入 run_id

✅ 参考答案

答案：A。 created_at 表示首次创建时间，不应在重跑时覆盖。updated_at、可变显示属性和最近批次通常在匹配或创建后更新，但也要遵守字段所有权。

10. 为什么应使用查询参数传入名称、来源和置信度？

A. 降低 Cypher 注入风险，并利于查询计划复用 \ B. 参数会自动修复所有脏数据 \ C. 参数可以替代唯一约束 \ D. 参数会自动创建标签

✅ 参考答案

答案：A。 数据值应通过参数传递，不应通过字符串拼接进入查询文本。参数化不能代替 Schema 验证、权限控制或业务白名单。

11. 为兼容不同 Neo4j 版本并控制 Schema，处理动态节点标签和关系类型最稳妥的课程方案是？

A. 用白名单把类型路由到预定义查询 \ B. 把用户输入直接拼到冒号后 \ C. 删除所有标签和关系类型 \ D. 对任意输入只做引号转义

✅ 参考答案

答案：A。 当前 Cypher 已提供部分动态标签/类型语法，但版本能力和查询结构需要明确管理。预定义查询加白名单最容易审计，也能阻止输入扩张图 Schema；数据属性仍用参数传递。

12. UNWIND $rows AS row 在批量导入中的作用是？

A. 将参数列表展开为查询中的多行，供后续 MATCH、MERGE、SET 处理 \ B. 保证所有行永久按输入顺序执行 \ C. 自动为每行创建独立数据库 \ D. 自动跳过所有非法行

✅ 参考答案

答案：A。 UNWIND 适合把一批结构相同的参数记录交给一个查询。数据验证、错误隔离和批大小仍由导入程序设计。

13. 若业务逻辑依赖批内顺序，关于 UNWIND 的正确判断是？

A. UNWIND 不保证输出行顺序，需显式 ORDER BY 或先消除顺序依赖 \ B. UNWIND 永远保持 Python 列表顺序 \ C. Neo4j 中所有查询都有默认稳定顺序 \ D. 只要用了 MERGE 就有顺序保证

✅ 参考答案

答案：A。 官方文档明确只有 ORDER BY 提供排序保证。建图任务通常应让每行独立且幂等，不把正确性绑定到隐式执行顺序。

14. 将一百万条关系放进单个事务的主要风险是？

A. 内存、锁持有、超时和失败重试成本过高 \ B. 唯一约束会失效 \ C. 参数化查询会变成字符串拼接 \ D. 图中不会产生关系

✅ 参考答案

答案：A。 应通过压测选择批大小，记录批次边界和失败行。批越大不一定越快；批越小也会增加往返开销。

15. Neo4j Driver 的托管事务可能因瞬时故障重试，因此事务回调应满足什么要求？

A. 幂等，且不依赖或重复修改不可控的外部全局状态 \ B. 每次重试都发送一封通知 \ C. 必须清空数据库 \ D. 必须生成随机节点 ID

✅ 参考答案

答案：A。 数据库事务本身保持原子性，但回调可能被再次执行。外部副作用、随机 ID 和非幂等计数会使重试产生不可预测结果。

16. 为什么通常要先创建唯一约束，再导入节点？

A. 尽早暴露重复身份，并在并发写入时守住唯一性 \ B. 约束会自动推断关系方向 \ C. 约束能替代输入验证 \ D. 约束创建后不再需要 ID

✅ 参考答案

答案：A。 先导入脏数据再建约束，约束可能因现存重复而失败。约束是数据库边界的最后一道防线，不是上游质检的替代品。

17. 一条关系的 to_id 在节点表中不存在时，最佳默认处理是？

A. 拒绝该关系并记录为缺失端点，待修复后重放 \ B. 只凭名称自动创建未知类型节点 \ C. 把关系改成自环 \ D. 忽略且计为成功

✅ 参考答案

答案：A。 悬空引用说明上游数据、导入顺序或 ID 映射有问题。静默造点会掩盖根因，并产生缺属性节点。

18. 为使“疾病推荐药品”可追溯，关系至少应携带什么？

A. 来源或证据 ID、抽取/导入版本及适用的置信信息 \ B. 仅关系颜色 \ C. Neo4j 内部元素 ID 即可 \ D. 浏览器窗口大小

✅ 参考答案

答案：A。 医学事实需要回答“来自哪里、何时、由哪个版本产生”。Neo4j 内部 ID 不等于可跨导入稳定的业务标识。

19. 两个可靠来源对同一药物推荐结论冲突时，图谱应如何处理？

A. 保留带来源的独立主张或证据，显式标记冲突和状态 \ B. 后导入者无痕覆盖前者 \ C. 随机保留一条 \ D. 将置信度都改为 1

✅ 参考答案

答案：A。 冲突是需要治理的信息。主张层可保留来源、时间、适用人群和审核状态，面向用户的回答再按政策聚合并提示不确定性。

20. 验证幂等性的最直接测试是？

A. 对同一快照连续导入两次，节点/关系业务键集合不变，第二次无意外新增 \ B. 只运行一次并查看界面 \ C. 每次导入前 DETACH DELETE 全库 \ D. 比较两次运行耗时是否相同

✅ 参考答案

答案：A。 还应检查 created_at 不被覆盖、可变字段按规则更新、审计计数合理。清库重建会掩盖幂等缺陷。

21. 新快照中某来源的一条主张消失时，安全的增量策略是？

A. 将该来源主张标为 retired/inactive，并保留批次和时间 \ B. 删除所有共享端点节点 \ C. 清空整个数据库 \ D. 不区分来源直接删除同语义全部关系

✅ 参考答案

答案：A。 “本来源不再提供”不等于事实绝对错误，也不应影响其他来源。软退役便于审计、恢复和时间查询。

22. 下列查询意图中，哪一个直接用于发现孤立点？

A. MATCH (n) WHERE NOT (n)--() RETURN n \ B. MATCH (n) RETURN count(n) \ C. MATCH ()-[r]->() RETURN type(r) \ D. SHOW INDEXES

✅ 参考答案

答案：A。 孤立点没有任何方向的关系。是否允许孤立要按标签和业务阶段定义，例如待链接概念可临时存在，但必须有明确阈值。

23. Disease-[HAS_SYMPTOM]->Drug 为什么应被质量规则拦截？

31. 原课件采用“名称作键、逐条字符串拼接、源码写密码”的方式。请给出迁移优先级。

✅ 参考答案

建议顺序：

  1. 冻结并备份当前数据，导出节点、关系和来源计数；
  2. 定义节点/关系中间层、稳定 ID、Schema 版本和字段所有权；
  3. 清洗重复并在空的影子数据库先创建唯一约束；
  4. 使用官方 Driver、密钥注入、参数化预定义查询与 UNWIND 分批导入；
  5. 执行幂等、对账、Schema、孤立点和溯源检查；

不能只把 py2neo 换成新 Driver 而保留不稳定身份与字符串拼接。

32. 设计一个最小但可审计的节点中间记录合同。

✅ 参考答案

至少包含 id、type、canonical_name、source_id、source_record_id、schema_version 和 status。可选 aliases、codes、valid_from、valid_to、attributes。应规定必填项、类型、字符规范、允许标签、ID 命名空间、空值策略及谁有权更新每个字段。患者级数据还需去标识化策略和访问分级。

33. 设计一个关系中间记录合同，说明关系业务键。

✅ 参考答案

至少包含 from_id、type、to_id、source_id、evidence_id、confidence、extractor_version、status 和时间字段。若同一事实允许多来源并存，业务键可用 from_id + type + to_id + source_id + evidence_id；若把事实与证据分层，则事实边用三元组键，证据建为独立 Claim/Evidence 节点。键的选择必须与冲突和退役策略一致。

34. 为什么推荐“Schema → 节点 → 关系 → 质量检查”的导入顺序？

✅ 参考答案

先建约束可在身份边界尽早失败；节点先入库使关系端点能通过稳定 ID 匹配；关系后写避免偷偷创建残缺端点；最后质量检查对实际提交结果做全量对账。每阶段均应产出输入数、去重数、成功数、失败数和检查点，失败时只重放受影响批次。

35. 创建唯一约束时发现存量重复 ID，应如何处理？

✅ 参考答案

先停止写入并导出冲突组，不要删除任意一条。比较来源、标准编码、属性和邻域，判断是重复、错误映射还是同一 ID 被错误复用；形成 winner、merge、split 或重分配方案，迁移关系和来源后再次对账。处理过程保留重定向与审计记录，然后创建约束并加入回归测试。

36. 分别写出节点、关系和缺失端点的 Cypher 设计要点。

✅ 参考答案

节点查询用 UNWIND 展开 rows，以标签 + id 进行 MERGE，created_at 放 ON CREATE，名称和批次等可变字段按所有权更新。关系查询先按各自标签和 id MATCH 两端，再 MERGE 固定类型关系并设置溯源属性。缺失端点不能由关系查询匿名创建；可先运行 anti-join 式检查，或比较写入前引用集合并把异常行送入隔离表。

37. 解释“MERGE 匹配完整模式”为什么容易被误用。

✅ 参考答案

MERGE 会尝试匹配它收到的整个模式，完整模式不存在时才整体创建。若把节点的可变名称、关系、端点和时间都塞入一个 MERGE，一处变化就可能导致模式不匹配，进而创建意外节点或触发唯一冲突。更清晰的做法是先分别 MERGE/MATCH 稳定身份节点，使变量绑定，再 MERGE 两个已绑定节点之间的关系。

38. 一个批次中第 731 条数据失败，如何兼顾事务原子性与可恢复性？

✅ 参考答案

先区分数据库事务失败与单行数据非法。每批作为原子事务，失败则整批回滚；记录 run_id、batch_id、输入摘要、异常类型和安全脱敏样例。确定是永久数据错误后可二分定位或预验证隔离坏行，修复后用同一业务键重放。检查点只在事务提交后推进，不能把“已发送”当作“已提交”。

39. 给出幂等导入的验收标准。

✅ 参考答案

同一快照连续执行两次后：

  • 节点和关系业务键集合、总数及来源计数不变；
  • 第二次 created 数为 0，matched/updated 计数符合设计；
  • created_at 不变，updated_at 是否变化符合书面约定；
  • 无新增重复 ID、孤立点、非法边和缺失溯源；
  • 失败重放与事务重试不会重复外部副作用；
  • 输出 manifest 可关联到相同源摘要与程序版本。

46. 实现 validate_spo(row)，校验关系中间记录的必填字段、类型白名单、置信度和来源；返回错误列表。

✅ 参考答案

ALLOWED = {
    "HAS_SYMPTOM": ("disease:", "symptom:"),
    "RECOMMEND_DRUG": ("disease:", "drug:"),
    "DO_EAT": ("disease:", "food:"),
    "NOT_EAT": ("disease:", "food:"),
}


def validate_spo(row):
    errors = []
    required = ("from_id", "type", "to_id", "source_id", "evidence_id")
    for key in required:
        if not isinstance(row.get(key), str) or not row[key].strip():
            errors.append("missing_or_invalid:" + key)

    relation_type = row.get("type")
    if relation_type not in ALLOWED:
        errors.append("invalid_relation_type")
    elif isinstance(row.get("from_id"), str) and isinstance(row.get("to_id"), str):
        from_prefix, to_prefix = ALLOWED[relation_type]
        if not row["from_id"].startswith(from_prefix):
            errors.append("invalid_from_type")
        if not row["to_id"].startswith(to_prefix):
            errors.append("invalid_to_type")

    confidence = row.get("confidence")
    if not isinstance(confidence, (int, float)) or isinstance(confidence, bool):
        errors.append("invalid_confidence")
    elif not 0.0 <= float(confidence) <= 1.0:
        errors.append("confidence_out_of_range")
    return sorted(set(errors))

    "to_id": "symptom:008",
    "source_id": "dataset:v1",

    "confidence": 0.92,
}

bad = dict(good, to_id="drug:009", confidence=1.2)

47. 实现 normalize_entity(raw)，只接受白名单类型和上游 source_id，并生成稳定业务 ID；名称变化不得改变 ID。

import re

TYPE_PREFIX = {
    "Disease": "disease",
    "Symptom": "symptom",
    "Drug": "drug",
    "Food": "food",
}


def normalize_entity(raw):
    entity_type = raw.get("type")
    if entity_type not in TYPE_PREFIX:
        raise ValueError("unsupported entity type")
    source_id = str(raw.get("source_id", "")).strip()
    if not re.fullmatch(r"[A-Za-z0-9._-]+", source_id):

    if not name:
        raise ValueError("empty name")
    return {
        "id": TYPE_PREFIX[entity_type] + ":" + source_id.lower(),
        "type": entity_type,
        "name": name,
        "source": str(raw.get("source", "")).strip(),
    }

assert a["name"] != b["name"]

48. 将一条疾病 JSON 转成去重、确定性排序的节点与关系载荷，并兼容旧字段 recommand_drug。

✅ 参考答案
Python
import hashlib
import json

FIELDS = {
    "symptom": ("Symptom", "HAS_SYMPTOM"),
    "recommand_drug": ("Drug", "RECOMMEND_DRUG"),
    "recommend_drug": ("Drug", "RECOMMEND_DRUG"),
    "do_eat": ("Food", "DO_EAT"),
    "not_eat": ("Food", "NOT_EAT"),
}


def stable_id(label, name):
    normalized = " ".join(name.split()).casefold()
    digest = hashlib.sha256((label + "\0" + normalized).encode("utf-8")).hexdigest()[:16]
    return label.lower() + ":" + digest


def build_payload(record, source_id):
    disease_name = " ".join(record["name"].split())
    disease_id = record.get("id") or stable_id("Disease", disease_name)
    nodes = {(disease_id, "Disease", disease_name)}
    relations = set()

    for field, (label, relation_type) in FIELDS.items():
        for raw_name in record.get(field, []) or []:
            name = " ".join(str(raw_name).split())
            if not name:
                continue
            target_id = stable_id(label, name)
            nodes.add((target_id, label, name))
            relations.add((disease_id, relation_type, target_id, source_id))

    node_rows = [
        {"id": node_id, "type": label, "name": name, "source": source_id}
        for node_id, label, name in sorted(nodes)
    ]
    relation_rows = [
        {"from_id": a, "type": r, "to_id": b, "source_id": s}
        for a, r, b, s in sorted(relations)
    ]
    return node_rows, relation_rows


raw = {
    "id": "disease:001",
    "name": "某疾病",
    "symptom": ["咳嗽", "咳嗽"],
    "recommand_drug": ["药品甲"],
    "recommend_drug": ["药品甲"],
}
nodes, relations = build_payload(raw, "medical:v1")
assert len(nodes) == 3
assert len(relations) == 2
assert json.dumps(nodes, ensure_ascii=False, sort_keys=True)
print(relations)

49. 实现 compile_upsert(kind, rows)，只允许白名单节点/关系类型，返回预定义参数化 Cypher 和参数；恶意类型必须被拒绝。

✅ 参考答案
Python
NODE_QUERIES = {
    "Disease": """
UNWIND $rows AS row
MERGE (n:Disease {id: row.id})
ON CREATE SET n.created_at = datetime()
SET n.name = row.name, n.source = row.source, n.updated_at = datetime()
""".strip(),
    "Symptom": """
UNWIND $rows AS row
MERGE (n:Symptom {id: row.id})
ON CREATE SET n.created_at = datetime()
SET n.name = row.name, n.source = row.source, n.updated_at = datetime()
""".strip(),
}

RELATION_QUERIES = {
    "HAS_SYMPTOM": """
UNWIND $rows AS row
MATCH (a:Disease {id: row.from_id})
MATCH (b:Symptom {id: row.to_id})
MERGE (a)-[r:HAS_SYMPTOM]->(b)
SET r.source_id = row.source_id, r.updated_at = datetime()
""".strip(),
}


def compile_upsert(kind, rows):
    if kind in NODE_QUERIES:
        query = NODE_QUERIES[kind]
    elif kind in RELATION_QUERIES:
        query = RELATION_QUERIES[kind]
    else:
        raise ValueError("kind is not allowlisted")
    if not isinstance(rows, list) or not rows:
        raise ValueError("rows must be a non-empty list")
    return query, {"rows": rows}


query, params = compile_upsert(
    "Disease",
    [{"id": "disease:001", "name": "某疾病", "source": "medical:v1"}],
)
assert "$rows" in query and params["rows"][0]["id"] == "disease:001"
try:
    compile_upsert("Disease) DETACH DELETE n //", [{}])
    raise AssertionError("malicious kind should fail")
except ValueError:
    pass
print("ok")

动态数据只进入 params；标签和关系类型由可审计白名单选择查询模板。

50. 实现 make_batches(rows, size, run_id)，按业务键确定性排序并为每批计算可复核摘要。

✅ 参考答案

import hashlib
import json


def canonical_bytes(value):
    text = json.dumps(
        value, ensure_ascii=False, sort_keys=True,
        separators=(",", ":")
    )
    return text.encode("utf-8")


def make_batches(rows, size, run_id):
    if not isinstance(size, int) or isinstance(size, bool) or size <= 0:
        raise ValueError("size must be a positive integer")
    ordered = sorted(rows, key=lambda row: (
        row.get("id", ""),
        row.get("from_id", ""),
        row.get("type", ""),
        row.get("to_id", ""),
        row.get("source_id", ""),
    ))
    batches = []
    for offset in range(0, len(ordered), size):
        payload = ordered[offset:offset + size]
        digest = hashlib.sha256(canonical_bytes(payload)).hexdigest()
        batches.append({
            "run_id": run_id,
            "batch_no": len(batches) + 1,
            "row_count": len(payload),
            "sha256": digest,
            "rows": payload,
        })
    return batches


rows = [{"id": "d:3"}, {"id": "d:1"}, {"id": "d:2"}]
first = make_batches(rows, 2, "kg-001")
second = make_batches(list(reversed(rows)), 2, "kg-001")
assert first == second
assert [b["row_count"] for b in first] == [2, 1]
print([(b["batch_no"], b["sha256"][:8]) for b in first])

摘要用于确认重放的是同一载荷；检查点只能在对应数据库事务提交成功后记录。

class MemoryGraph:
    def __init__(self):
        self.nodes = {}
        self.relations = {}

    def upsert_node(self, row, run_id):
        key = (row["type"], row["id"])
        if key not in self.nodes:
            self.nodes[key] = {
                "id": row["id"],
                "type": row["type"],
                "created_run": run_id,
            }
        node = self.nodes[key]
        node["name"] = row["name"]
        node["source"] = row["source"]
        node["updated_run"] = run_id

    def upsert_relation(self, row, run_id):
        key = (
            row["from_id"], row["type"], row["to_id"],
            row["source_id"], row["evidence_id"],
        )
        if key not in self.relations:
            self.relations[key] = {
                "created_run": run_id,
                "key": key,
            }
        relation = self.relations[key]
        relation["confidence"] = row["confidence"]
        relation["updated_run"] = run_id

52. 实现 audit_graph(nodes, relations)，检查重复节点键、缺失端点、非法 domain/range、自环和缺失来源。

✅ 参考答案
Python
from collections import Counter

SCHEMA = {
    "HAS_SYMPTOM": ("Disease", "Symptom"),
    "RECOMMEND_DRUG": ("Disease", "Drug"),
    "DO_EAT": ("Disease", "Food"),
    "NOT_EAT": ("Disease", "Food"),
}


def audit_graph(nodes, relations):
    keys = [(node.get("type"), node.get("id")) for node in nodes]
    counts = Counter(keys)
    duplicates = sorted(key for key, count in counts.items() if count > 1)
    node_types = {node.get("id"): node.get("type") for node in nodes}

    missing_endpoints = []
    schema_violations = []
    self_loops = []
    missing_source = []
    for index, relation in enumerate(relations):
        left, right = relation.get("from_id"), relation.get("to_id")
        relation_type = relation.get("type")
        if left not in node_types or right not in node_types:
            missing_endpoints.append(index)
        expected = SCHEMA.get(relation_type)
        actual = (node_types.get(left), node_types.get(right))
        if expected is None or actual != expected:
            schema_violations.append(index)
        if left == right:
            self_loops.append(index)
        if not relation.get("source_id"):
            missing_source.append(index)
    return {
        "duplicate_node_keys": duplicates,
        "missing_endpoints": missing_endpoints,
        "schema_violations": schema_violations,
        "self_loops": self_loops,
        "missing_source": missing_source,
    }


nodes = [
    {"id": "d:1", "type": "Disease"},
    {"id": "s:1", "type": "Symptom"},
    {"id": "s:1", "type": "Symptom"},
]
relations = [
    {"from_id": "d:1", "type": "HAS_SYMPTOM",
     "to_id": "s:1", "source_id": "v1"},
    {"from_id": "d:1", "type": "RECOMMEND_DRUG",
     "to_id": "x:1", "source_id": ""},
]
report = audit_graph(nodes, relations)
assert report["duplicate_node_keys"] == [("Symptom", "s:1")]
assert report["missing_endpoints"] == [1]
assert report["schema_violations"] == [1]
assert report["missing_source"] == [1]
print(report)

53. 实现 validate_plan(steps)，要求四阶段顺序为 constraints、nodes、relations、checks，检查点编号连续且每步输入摘要非空。

✅ 参考答案

EXPECTED = ["constraints", "nodes", "relations", "checks"]


def validate_plan(steps):
    errors = []
    phases = [step.get("phase") for step in steps]
    if phases != EXPECTED:
        errors.append("invalid_phase_order")

    checkpoints = [step.get("checkpoint") for step in steps]
    if checkpoints != list(range(1, len(steps) + 1)):
        errors.append("invalid_checkpoints")

    run_ids = {step.get("run_id") for step in steps}
    if len(run_ids) != 1 or None in run_ids or "" in run_ids:
        errors.append("inconsistent_run_id")

    for index, step in enumerate(steps):
        digest = step.get("input_sha256")
        if not isinstance(digest, str) or len(digest) != 64:
            errors.append("invalid_digest_at:" + str(index))
    return errors

54. 实现 diff_claims(old_rows, new_rows)，按来源级主张业务键计算 upsert 与 retire，不做物理删除。

✅ 参考答案
Python
def claim_key(row):
    return (
        row["from_id"],
        row["type"],
        row["to_id"],
        row["source_id"],
        row["evidence_id"],
    )


def diff_claims(old_rows, new_rows):
    old = {claim_key(row): dict(row) for row in old_rows}
    new = {claim_key(row): dict(row) for row in new_rows}
    upsert = []
    for key in sorted(new):
        if key not in old or new[key] != old[key]:
            row = dict(new[key])
            row["status"] = "active"
            upsert.append(row)

    retire = []
    for key in sorted(set(old) - set(new)):
        row = dict(old[key])
        row["status"] = "retired"
        retire.append(row)
    return upsert, retire


base = {
    "from_id": "d:1", "type": "HAS_SYMPTOM", "to_id": "s:1",
    "source_id": "medical:v1", "evidence_id": "line:1",
    "confidence": 0.8,
}
removed = dict(base)
changed = dict(base, evidence_id="line:2", confidence=0.9)
upsert, retire = diff_claims([removed], [changed])
assert [row["status"] for row in upsert] == ["active"]
assert [row["status"] for row in retire] == ["retired"]
assert "status" not in removed
print({"upsert": upsert, "retire": retire})

退役范围由 source_id + evidence_id 限定，不会删除其他来源对同一三元组的主张。最后一个断言也验证输入记录没有被函数原地修改。

55. 实现 quality_gate(stats, limits)，计算完整率、孤立率、溯源覆盖率和重复率，并返回是否放行及失败原因。

✅ 参考答案
Python
def safe_ratio(numerator, denominator):
    return numerator / denominator if denominator else 0.0


def quality_gate(stats, limits):
    nodes = stats["nodes"]
    relations = stats["relations"]
    metrics = {
        "node_completeness": 1.0 - safe_ratio(
            stats["nodes_missing_required"], nodes
        ),
        "isolated_rate": safe_ratio(stats["isolated_nodes"], nodes),
        "provenance_coverage": 1.0 - safe_ratio(
            stats["relations_missing_source"], relations
        ),
        "duplicate_rate": safe_ratio(stats["duplicate_keys"], nodes),
        "schema_violation_rate": safe_ratio(
            stats["schema_violations"], relations
        ),
    }
    failures = []
    for metric in ("node_completeness", "provenance_coverage"):
        if metrics[metric] < limits["min_" + metric]:
            failures.append(metric + "_below_min")
    for metric in ("isolated_rate", "duplicate_rate", "schema_violation_rate"):
        if metrics[metric] > limits["max_" + metric]:
            failures.append(metric + "_above_max")
    return {"passed": not failures, "metrics": metrics, "failures": failures}


stats = {
    "nodes": 1000,
    "relations": 3000,
    "nodes_missing_required": 0,
    "isolated_nodes": 5,
    "relations_missing_source": 0,
    "duplicate_keys": 0,
    "schema_violations": 1,
}
limits = {
    "min_node_completeness": 1.0,
    "min_provenance_coverage": 1.0,
    "max_isolated_rate": 0.01,
    "max_duplicate_rate": 0.0,
    "max_schema_violation_rate": 0.0,
}
result = quality_gate(stats, limits)
assert result["passed"] is False
assert result["failures"] == ["schema_violation_rate_above_max"]
print(result)

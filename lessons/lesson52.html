🏷️ 数据标注、查询语言与 Neo4j 工具链

🏷️ Doccano🏷️ Cypher🏷️ Neo4j✏️ 配套练习：数据标注、查询语言与 Neo4j 工具链综合训练🚀 配套项目：医学知识图谱：从信息抽取到 KBQA

数据标注、查询语言与 Neo4j 工具链

本讲整合原课件“项目工具”四节，把安装步骤改组为“标注质量—图模型—查询实践”的复习链路。

学习目标

  • 用 Doccano 组织实体和关系标注；
  • 设计标签、标注指南、数据划分与一致性检查；
  • 理解 SQL、SPARQL、Gremlin 和 Cypher 的主要差异；
  • 使用 Cypher 创建、合并、查询和删除图数据。

Doccano 在项目中的位置

Doccano 是标注平台，不是标注规范本身。平台负责上传语料、管理标签、分配成员和导出结果；项目团队仍然要决定“标什么、如何标、冲突怎么裁决”。

一份可用的标注指南

要素 需要写清的内容
任务定义 是实体标注、关系标注还是文本分类
标签定义 Disease、Symptom、Drug、Food 各指什么
边界规则 是否包括修饰语、程度词、否定词和单位
嵌套与重叠 允许标注哪一层，是否允许一段文本多标签
关系方向 统一使用 Disease -> Symptom 而不是随文本语序变化
负样本 否定、假设、家族史和过去史是否入图
困难案例 至少列出 10–20 个边界例子及裁决结果

推荐的标注流程

  1. 随机抽取一小批样本，两名标注员独立试标；
  2. 统计标签分布和标注员一致性，整理分歧；
  3. 由业务专家裁决边界案例，回写指南；
  4. 标注员通过一致性门槛后再扩大标注；
  5. 导出后做 Schema 校验、边界校验、重复样本和泄漏检查。

随机切分前应先去重或分组。同一来源文档的重复句如果同时出现在训练集与测试集，会虚高评估结果。

四种查询语言的角色

语言 主要数据模型 查询风格 本项目中的位置
SQL 关系表 表、连接、聚合 运营表、标注任务和日志可使用
SPARQL RDF 三元组 图模式匹配 需与标准本体 / Linked Data 交互时常用
Gremlin 属性图 遍历步骤链 面向多种图引擎的通用遍历思维
Cypher 属性图 ASCII 图模式 Neo4j 中的主要查询语言

Cypher 用圆括号表示节点，方括号表示关系，箭头表示方向：

CYPHER
(d:Disease)-[:HAS_SYMPTOM]->(s:Symptom)

Neo4j 中的节点、关系与属性

  • Label 表示节点类型，如 Disease；
  • Relationship Type 表示关系语义，如 HAS_SYMPTOM；
  • Property 存储标量或列表值，如 name、source；
  • 约束用来防止重复标识，索引用来加快定位。
CYPHER
CREATE CONSTRAINT disease_id_unique IF NOT EXISTS
FOR (d:Disease) REQUIRE d.id IS UNIQUE;

CREATE INDEX disease_name_index IF NOT EXISTS
FOR (d:Disease) ON (d.name);

早期 Neo4j 课件中的 CREATE CONSTRAINT ON ... ASSERT ... 属于旧语法。新项目应使用 FOR ... REQUIRE ... 形式，并在复现时锁定 Neo4j 主版本。

Cypher 基本操作

CREATE 与 MERGE

CREATE 每次都创建新数据；MERGE 按给定模式匹配，不存在时才创建。可重跑的导入脚本通常以 MERGE 为主。

CYPHER
MERGE (d:Disease {id: $disease_id})
SET d.name = $name,
    d.updated_at = datetime();

MATCH、WHERE 与 RETURN

CYPHER
MATCH (d:Disease)-[:HAS_SYMPTOM]->(s:Symptom)
WHERE d.id = $disease_id
RETURN d.name AS disease, collect(DISTINCT s.name) AS symptoms
ORDER BY disease;

建立关系

CYPHER
MATCH (d:Disease {id: $disease_id})
MATCH (s:Symptom {id: $symptom_id})
MERGE (d)-[r:HAS_SYMPTOM]->(s)
SET r.source = $source,
    r.confidence = $confidence;

删除的区别

CYPHER
// 节点还有关系时，DELETE 会失败
MATCH (n:Symptom {id: $id}) DELETE n;

// 同时删除该节点相连的关系
MATCH (n:Symptom {id: $id}) DETACH DELETE n;

在导入程序中不应把“清空整库”当成默认前置条件。先约束唯一 ID，再用 MERGE 实现幂等更安全。

参数化是必须项

不要把用户输入或节点名称拼进 Cypher 字符串。参数化能避免引号错误与查询注入，也更利于执行计划复用。

Python
cypher = """
MATCH (d:Disease {id: $disease_id})-[:HAS_SYMPTOM]->(s:Symptom)
RETURN collect(DISTINCT s.name) AS symptoms
"""
record = driver.execute_query(
    cypher,
    disease_id="disease:000123",
    database_="neo4j",
).records[0]

Label 和 Relationship Type 不能像普通属性值一样直接参数化。如果业务需要动态类型，应从程序内部白名单选择，而不是接受任意用户输入。

工具边界

  • Flask / FastAPI 负责 HTTP 接口，不应直接暴露 Neo4j 凭据；
  • Gunicorn 是常见 WSGI 进程管理器，不能代替超时、认证、限流与监控；
  • Neo4j Browser 适合开发调试，不等于面向终端用户的产品界面；
  • 数据库密码、API Key 与模型路径应通过环境变量或密钥服务注入。

本讲小结

  1. 标注平台只解决流程，标注指南与质检才决定数据质量。
  2. Cypher 面向属性图的模式匹配；唯一约束 + MERGE 是幂等建图的基础。
  3. 查询值必须参数化，动态 Label / Relationship Type 必须走内部白名单。

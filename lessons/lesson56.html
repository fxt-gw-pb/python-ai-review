Neo4j 医学图谱搭建与质量验证

本讲对应原课件第六章，保留“解析 JSON—提取三元组—写入节点与关系—图展示”主线，将旧式逐条字符串拼接升级为唯一约束、参数化批处理和幂等导入。

学习目标

  • 将原始 JSON 规范化为节点与关系记录；
  • 在导入前创建唯一约束和必要索引；
  • 使用 Neo4j 官方 Python Driver、参数化 Cypher 和 UNWIND 批量导入；
  • 使用 MERGE 实现可重复运行的建图管道；
  • 用孤立点、悬空边、重复和属性完整率验收。

输入与中间层

原课件数据以一行一个 JSON 的疾病记录为主：

JSON
{
  "name": "某疾病",
  "symptom": ["症状A", "症状B"],
  "recommand_drug": ["药品A"],
  "do_eat": ["食物A"],
  "not_eat": ["食物B"]
}

原字段中的 recommand_drug 是历史拼写。上游兼容可保留该字段，但中间层应统一为 recommend_drug，避免把历史错误传递到图 Schema 和 API。

推荐先产出两类中间记录：

JSON
{"id": "disease:001", "type": "Disease", "name": "某疾病", "source": "doc:001"}
{"from_id": "disease:001", "type": "HAS_SYMPTOM", "to_id": "symptom:008", "source": "doc:001", "confidence": 1.0}

先在文件层验证中间记录，会比把解析逻辑与数据库写入紧耦合更容易测试和回滚。

第一步：建立约束和索引

CYPHER
CREATE CONSTRAINT disease_id_unique IF NOT EXISTS
FOR (n:Disease) REQUIRE n.id IS UNIQUE;

CREATE CONSTRAINT symptom_id_unique IF NOT EXISTS
FOR (n:Symptom) REQUIRE n.id IS UNIQUE;

CREATE CONSTRAINT drug_id_unique IF NOT EXISTS
FOR (n:Drug) REQUIRE n.id IS UNIQUE;

CREATE CONSTRAINT food_id_unique IF NOT EXISTS
FOR (n:Food) REQUIRE n.id IS UNIQUE;

CREATE INDEX disease_name_index IF NOT EXISTS
FOR (n:Disease) ON (n.name);

先建约束再导入。如果导入数据已存在重复 ID，约束会让错误在入库边界显式暴露。

第二步：批量合并节点

CYPHER
UNWIND $rows AS row
MERGE (n:Disease {id: row.id})
ON CREATE SET n.created_at = datetime()
SET n.name = row.name,
    n.aliases = coalesce(row.aliases, []),
    n.source = row.source,
    n.updated_at = datetime();

MERGE 只应包含稳定键。如果把 name、source、updated_at 都放进 MERGE 模式，任意属性变化都可能生成新节点。

第三步：批量合并关系

CYPHER
UNWIND $rows AS row
MATCH (d:Disease {id: row.from_id})
MATCH (s:Symptom {id: row.to_id})
MERGE (d)-[r:HAS_SYMPTOM]->(s)
SET r.source = row.source,
    r.confidence = row.confidence,
    r.updated_at = datetime();

每一类关系使用固定的参数化查询。不要把 row.type 直接拼到关系类型中；如确有多类关系，应在程序中根据白名单路由到预定义查询。

官方 Python Driver 示例

Python
import os
from neo4j import GraphDatabase

URI = os.environ["NEO4J_URI"]
AUTH = (os.environ["NEO4J_USER"], os.environ["NEO4J_PASSWORD"])

NODE_QUERY = """
UNWIND $rows AS row
MERGE (n:Disease {id: row.id})
SET n.name = row.name,
    n.source = row.source,
    n.updated_at = datetime()
"""

def import_diseases(rows: list[dict]) -> None:
    with GraphDatabase.driver(URI, auth=AUTH) as driver:
        driver.verify_connectivity()
        driver.execute_query(
            NODE_QUERY,
            rows=rows,
            database_="neo4j",
        )

生产导入还应补充：批大小、事务重试、失败批记录、总数对账和日志脱敏。密码不应写死在源码中。

为什么不默认 delete_all()

清空整库再导入虽然简单，但会：

  • 丢失其他数据源已入图的知识；
  • 让每次发布都出现不可用窗口；
  • 掩盖 ID 不稳定或幂等逻辑错误；
  • 难以审计哪些节点是新增、更新或删除。

开发测试可使用独立数据库或容器重建；持久环境应使用增量、可审计的导入策略。

图质量验收查询

各类节点和关系数量

CYPHER
MATCH (n)
UNWIND labels(n) AS label
RETURN label, count(*) AS nodes
ORDER BY nodes DESC;
CYPHER
MATCH ()-[r]->()
RETURN type(r) AS relation, count(*) AS edges
ORDER BY edges DESC;

缺失标准 ID 或名称

CYPHER
MATCH (n)
WHERE n.id IS NULL OR n.name IS NULL OR trim(n.name) = ""
RETURN labels(n) AS labels, count(*) AS bad_nodes;

孤立点

CYPHER
MATCH (n)
WHERE NOT (n)--()
RETURN labels(n) AS labels, count(*) AS isolated_nodes;

名称冲突候选

CYPHER
MATCH (n)
WITH labels(n) AS labels, n.name AS name, collect(n.id) AS ids
WHERE size(ids) > 1
RETURN labels, name, ids
ORDER BY size(ids) DESC;

超出 Schema 的边

CYPHER
MATCH (a)-[r]->(b)
WHERE NOT (
  (a:Disease AND type(r) = "HAS_SYMPTOM" AND b:Symptom) OR
  (a:Disease AND type(r) = "RECOMMEND_DRUG" AND b:Drug) OR
  (a:Disease AND type(r) IN ["DO_EAT", "NOT_EAT"] AND b:Food)
)
RETURN labels(a), type(r), labels(b), count(*) AS violations;

运行与版本化

每次导入建议产生一份 manifest：

JSON
{
  "run_id": "kg-20260716-001",
  "schema_version": "1.0.0",
  "source_snapshot": "sha256:...",
  "extractor_version": "ner-1.2.0+re-1.1.0",
  "node_count": 0,
  "edge_count": 0,
  "failed_rows": 0
}

这使一条错误回答能追溯到具体数据快照、抽取模型和建图批次。

本讲小结

  1. 先生成可验证的中间节点 / 关系记录，再与数据库对接。
  2. 唯一约束、稳定 ID、参数化 UNWIND 和 MERGE 是可重跑建图的核心。
  3. 图谱不能以“能展示”为验收终点，必须统计重复、缺失、孤立点和 Schema 违反。

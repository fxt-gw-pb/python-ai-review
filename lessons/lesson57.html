医疗 KBQA：NLU、Cypher 编排与可追溯回答

本讲对应原课件第七章：问答系统基础、意图识别与槽位填充、医疗 KBQA 架构及实现。本版增加参数化查询、返回合同、可追溯证据、澄清和安全拒答。

学习目标

  • 区分检索式、生成式与混合问答；
  • 将医疗问题解析为意图、槽位和已链接实体；
  • 理解 NLU、DM、DST 与 PL 各模块的输入输出；
  • 用白名单查询模板和参数化 Cypher 访问 Neo4j；
  • 实现澄清、未命中、拒答、来源展示与端到端评估。

问答系统类型

类型 答案来源 优势 主要风险
检索式 数据库或文档原文 可追溯、输出稳定 覆盖受知识库限制
模板式 KBQA 图谱查询结果 + 回复模板 语义和输出可控 意图与模板扩展成本高
生成式 模型参数与上下文 表达自然、泛化强 可能幻觉，格式与来源不稳定
混合式 图谱 + 文档 + 生成模型 覆盖和表达能力更强 编排、冲突与评估更复杂

课件项目的主线是模板式 KBQA，很适合用来学习语义解析、图查询和会话管理的模块边界。

系统架构

Text
用户问题
  → 输入校验与安全路由
  → NLU：意图识别 + 槽位填充 / NER + 实体链接
  → DM：置信度策略、澄清、兜底与对话状态
  → PL：选择查询模板并绑定参数
  → Neo4j：执行参数化 Cypher
  → 答案构建：内容 + 来源 + 时间 + 状态
  → 日志、评估与反馈

原课件使用的缩写：

  • NLU（Natural Language Understanding）：理解当前输入；
  • DM（Dialogue Management）：管理对话策略与转移；
  • DST（Dialogue State Tracking）：保留当前轮次已确认的意图、槽位和澄清状态；
  • PL（Policy / Planning Layer）：将结构化语义映射为查询或动作。

意图与槽位

“慢性支气管炎有哪些症状？”可解析为：

JSON
{
  "intent": {"name": "disease_symptoms", "confidence": 0.94},
  "slots": [
    {
      "name": "disease",
      "mention": "慢性支气管炎",
      "entity_id": "disease:000123",
      "confidence": 0.98
    }
  ]
}

意图识别方法可以是规则模板、传统文本分类、BERT 类模型或大模型结构化输出。槽位填充可用规则、BiLSTM-CRF、Transformer NER，但 NER 的 mention 仍需实体链接才能安全查询。

查询模板白名单

不要让用户文本直接生成并执行任意 Cypher。对于封闭问答范围，将意图映射到已审核模板：

Python
QUERY_TEMPLATES = {
    "disease_symptoms": """
        MATCH (d:Disease {id: $disease_id})-[:HAS_SYMPTOM]->(s:Symptom)
        RETURN d.name AS disease,
               collect(DISTINCT s.name) AS values,
               collect(DISTINCT s.source) AS sources
    """,
    "disease_drugs": """
        MATCH (d:Disease {id: $disease_id})-[:RECOMMEND_DRUG]->(x:Drug)
        RETURN d.name AS disease,
               collect(DISTINCT x.name) AS values,
               collect(DISTINCT x.source) AS sources
    """,
}

绑定参数时使用实体 ID，而不是原始 mention：

Python
query = QUERY_TEMPLATES[semantic_frame["intent"]["name"]]
records, summary, keys = driver.execute_query(
    query,
    disease_id=semantic_frame["slots"][0]["entity_id"],
    database_="neo4j",
)

对话策略不是两个阈值就结束

基础策略可以按置信度分为接受、澄清和拒绝：

Python
def decide(intent_conf: float, link_conf: float, has_slot: bool) -> str:
    if not has_slot:
        return "ask_missing_slot"
    if min(intent_conf, link_conf) >= 0.80:
        return "execute"
    if min(intent_conf, link_conf) >= 0.45:
        return "clarify"
    return "fallback"

但生产策略还要检查：

  • 是否命中支持的意图和实体类型；
  • 实体链接是否为 NIL 或存在多个接近候选；
  • 图查询是“实体不存在”还是“实体存在但暂无该关系”；
  • 数据是否过期或来源不符合当前场景；
  • 问题是否涉及诊断、个体化用药或紧急状况。

显式的对话状态

不要用一个全局 JSON 文件保存所有用户的澄清状态。每个会话应有隔离的状态记录：

JSON
{
  "session_id": "opaque-id",
  "pending_intent": "disease_symptoms",
  "pending_entity_id": "disease:000123",
  "pending_question": "您指的是慢性支气管炎吗？",
  "expires_at": "2026-07-16T12:10:00Z"
}

状态需要过期时间、用户 / 会话隔离和并发安全。必要时还应加密或避免存储敏感原文。

答案返回合同

一个可调试、可追溯的 API 不只返回一段文本：

JSON
{
  "status": "answered",
  "answer": "图谱中收录的相关症状包括……",
  "intent": "disease_symptoms",
  "entities": [{"id": "disease:000123", "name": "慢性支气管炎"}],
  "evidence": [{"source": "doc:2026-001", "updated_at": "2026-06-01"}],
  "confidence": 0.91,
  "trace_id": "request-opaque-id",
  "disclaimer": "仅用于知识检索，不替代专业医疗建议。"
}

status 可以是 answered、clarification_needed、not_found、unsupported、unsafe 或 system_error。不同状态不应都包装成自信的自然语言答案。

端到端评估

层级 主指标 常见失败
意图 Macro / Micro F1、校准误差 相似问法混淆、置信度过度自信
槽位 / NER 实体级 P/R/F1 边界错、类型错、否定范围错
实体链接 Accuracy、MRR、NIL F1 同名实体误链接、新概念被强制映射
查询 模板选择准确率、执行成功率 查询模板错、参数错、Schema 漂移
答案 Exact Match / 集合 F1、追溯覆盖率 漏项、重复、来源缺失
安全 应拒答准确率、紧急问题路由率 超范围推断、诊断或用药建议

端到端测试集需要包含标准问法、口语改写、错别字、多实体、少槽位、歧义实体、未收录问题和安全负例。

医疗安全底线

安全提醒：本系统只展示图谱中已审核的知识，不应根据个人症状做诊断，不应提供剂量、处方或替代就医的建议。遇到胸痛、呼吸困难、意识障碍等可能的紧急情况，应立即停止普通问答，提示联系当地急救服务或尽快就医。

本讲小结

  1. KBQA 不是一条“问题到 Cypher”的直线，而是意图、槽位、链接、策略、查询和证据的状态机。
  2. 封闭域问答应使用白名单查询模板与参数化实体 ID，不执行用户可控的任意 Cypher。
  3. 回答必须携带状态、证据和时间，并把澄清、未命中与安全拒答当成正常能力。

🕸️ 知识图谱基础与医学图谱架构

🏷️ 图模型 / 三元组🏷️ Schema🏷️ 技术架构✏️ 配套练习：知识图谱基础与医学图谱架构综合训练🚀 配套项目：医学知识图谱：从信息抽取到 KBQA

知识图谱基础与医学图谱架构

本讲根据《项目简介》与《知识图谱技术概括》重组：保留原课件的分层与建图主线，补充 Schema、来源追踪和可验收指标。

学习目标

  • 用实体、属性、关系和三元组解释知识图谱；
  • 区分本体 / Schema 与具体图数据；
  • 说清数据获取、信息抽取、知识融合、知识存储和应用的边界；
  • 为一个最小医学图谱定义可验收的范围。

什么是知识图谱

知识图谱用“节点 + 边 + 属性”表达现实世界的对象与联系。最常用的基本单元是三元组：

Text
(主体, 谓词, 客体)
(慢性支气管炎, has_symptom, 咳嗽)
(慢性支气管炎, recommend_drug, 某药品)
概念 含义 医学例子
实体 可独立识别的对象 疾病、症状、药品、食物
实体类型 节点所属的类 Disease、Symptom
关系 两个实体间的有向语义 疾病 has_symptom 症状
属性 对实体或关系的描述 标准名、别名、来源、置信度
实体 ID 不随名称变化的唯一标识 disease:000123

同一条知识不应只有“名称字符串”。在可追溯系统中，还需要保留来源文档、发布时间、抽取方法和置信度。

Schema 与实例数据

Schema 回答“图谱允许有什么”，实例数据回答“图谱当前有什么”。

Text
Schema:
  (Disease)-[:HAS_SYMPTOM]->(Symptom)
  (Disease)-[:RECOMMEND_DRUG]->(Drug)
  (Disease)-[:DO_EAT]->(Food)
  (Disease)-[:NOT_EAT]->(Food)

Instance:
  (慢性支气管炎)-[:HAS_SYMPTOM]->(咳嗽)

一个关系定义至少要写清：

  1. 主体类型和客体类型；
  2. 方向和语义；
  3. 是否允许多值、自环和重复；
  4. 来源、时间与冲突的处理规则。

两种建图方式

自顶向下

先由业务专家定义 Schema，再按约束导入数据。适合安全性、一致性要求高的医学场景。优点是语义稳定、便于治理；代价是前期专家成本高。

自底向上

先从大量文本中抽取实体和关系，再归纳概念与模式。适合快速扩展和未知概念发现，但容易引入冗余、歧义和错误关系。

实际项目常用混合路线：核心 Schema 自顶向下，长尾实体和候选关系自底向上发现，经校验后入图。

医学图谱的技术链路

Text
可信数据源
  → 解析与清洗
  → 实体抽取 NER
  → 关系抽取 RE
  → 实体统一 / 链接 / 消歧
  → 质量校验与人工复核
  → 图数据库存储
  → 查询、问答与下游分析
阶段 核心问题 不能只看的指标
数据获取 来源是否可信、有权使用、可追溯？ 文档数量
信息抽取 是否找对实体边界、类型和关系？ Token Accuracy
知识融合 “高血压”和“原发性高血压”是否同一概念？ 字符串相等率
存储治理 是否有重复节点、悬空边和无来源知识？ 节点数
应用 回答是否正确、可追溯、能在不确定时拒答？ 演示是否能跑

最小医学图谱范围

课件项目以四类实体和四类关系构成闭环：

  • 节点：Disease、Symptom、Drug、Food；
  • 关系：HAS_SYMPTOM、RECOMMEND_DRUG、DO_EAT、NOT_EAT；
  • 疾病属性：标准名、别名、简介、来源、更新时间；
  • 问答范围：定义、症状、推荐药品、饮食宜忌。

这个范围能验证全部工程链路，却不会把“疾病定义”错误扩张为临床诊断。

质量与安全边界

安全提醒：医学知识图谱的演示项目只能用于知识检索与 NLP 学习，不构成诊断、处方或个体化用药建议。数据过期、来源不明、实体歧义或查询未命中时，系统应显式说明不确定性并引导用户寻求专业帮助。

本讲小结

  1. 图谱的核心不是可视化，而是稳定、可查询的语义模型。
  2. Schema 先约束实体、关系和方向，实例数据才有一致性。
  3. 医学图谱必须同时评估抽取质量、图质量、问答质量和安全边界。

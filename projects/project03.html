MULTI-AGENT TRAVEL ASSISTANT PROJECT

从自然语言旅行需求走到可路由、可协作、可执行的多智能体系统

SmartVoyage 面向天气查询、火车与飞机票务、演唱会票务和订票等复合需求：主控 Agent 负责理解意图与编排任务，专业 Agent 通过 A2A 协作，每个 Agent 再经 MCP 调用数据库或外部 API，最后由 Streamlit 向用户交付统一结果。

核心任务多智能体旅行助手理解需求 · 路由协作 · 调用工具 · 汇总回答

项目不仅讲解 Agent 概念，还给出协议、服务和完整工程实现：从配置、日志、MySQL 与天气数据采集开始，逐步构建天气、票务、订票 MCP Server 与 A2A Agent，最终完成意图路由、主控编排和 Web 交互。

三层调用机制如何分工

机制解决的问题交互双方在项目中的位置
Function Call把模型决策转换为结构化函数名与参数LLM ↔ 应用内工具单个 Agent 选择并调用具体能力
MCP用统一协议暴露工具、资源与提示MCP Client ↔ MCP Server天气、票务和订票工具服务层
A2A发现远程 Agent、传递任务并交换产物Client Agent ↔ Server Agent主控与专业 Agent 的跨服务协作层

五种 Agent 模式的演进

模式核心能力适用任务
工具使用选择并调用一个外部工具明确、单步的查询或执行
ReActThought / Action / Observation 循环需要观察结果后动态调整的任务
反思评估结果并自我修正质量要求高、允许迭代的产出
规划先拆解复杂目标，再分步执行长链路、多约束任务
多智能体按角色分工、通信与协作跨领域、可并行或需专业分工的系统
🛡️

工程边界：Agent 能调用工具并不等于可以无约束执行。涉及订票、支付、个人信息或外部写操作时，应加入参数校验、权限控制、幂等键、人工确认、超时重试、审计日志和失败补偿；对 ReAct 的内部推理只保留必要的结构化执行轨迹。

推荐复习方式

  1. 先用 Function Call 建立“模型只负责决策，程序负责执行”的边界，再学习 MCP 与 A2A。
  2. 分别跑通天气、票务和订票工具服务，并用固定输入验证参数、返回结构和错误处理。
  3. 在独立服务可用后再接入 A2A Agent，先验证单 Agent，再验证串行协作和意图路由。
  4. 对查询类操作设置超时、重试和缓存；对订票等写操作增加确认、幂等和审计。
  5. 最后接入主控 Agent 与 Streamlit，用正常问题、歧义问题、多意图问题和失败场景共同验收。
